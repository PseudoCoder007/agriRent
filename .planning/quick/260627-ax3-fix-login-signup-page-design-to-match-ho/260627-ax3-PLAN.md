<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client → forgot-password Server Action | Untrusted email input crosses into requestPasswordResetAction |
| Client → reset-password Server Action | Untrusted new-password input crosses into updatePasswordAction; the action trusts the caller's Supabase session (set by the recovery code exchange) to scope the update to the correct user |
| Supabase recovery email link → /auth/reset-callback | Untrusted `code` query param crosses into exchangeCodeForSession |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-ax3-01 | Information Disclosure | requestPasswordReset() | mitigate | Always return the same generic success message regardless of whether the email exists, mirroring logIn()'s existing anti-enumeration pattern; only a rate-limit error gets a distinct message |
| T-ax3-02 | Information Disclosure | requestPasswordReset()/updatePassword() error paths | mitigate | Never forward raw Supabase error.message to the client; log server-side via console.error, return only the predefined generic strings |
| T-ax3-03 | Spoofing | /auth/reset-callback | accept | Relies on Supabase's own PKCE code-exchange validity window/signature; no additional app-level check needed since exchangeCodeForSession itself rejects invalid/expired codes |
| T-ax3-04 | Tampering | reset-password page | mitigate | updatePassword() runs supabase.auth.updateUser() against the session established by the code exchange (server-set cookie), not a client-supplied user id — a user cannot target another account's password |
</threat_model>
