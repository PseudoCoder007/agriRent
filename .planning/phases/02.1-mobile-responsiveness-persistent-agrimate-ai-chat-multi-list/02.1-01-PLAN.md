<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Server Action | Untrusted `role`/`content` input crosses into `saveChatMessageAction` from the client chat widget |
| Server Action -> Postgres | `chat.service.ts` queries scoped by `userId` taken from the server-verified session, never from client input |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-01 | Information Disclosure | chat_messages RLS | mitigate | `user_id = (select auth.uid())` SELECT policy enforced at the database layer, not just filtered in the Server Action -- a forged userId cannot read another user's history even if the action layer had a bug |
| T-02.1-02 | Denial of Service | saveChatMessageSchema content field | mitigate | `.max(2000)` length cap rejects oversized payloads before they reach the database |
| T-02.1-03 | Tampering | saveChatMessageAction role parameter | mitigate | `z.enum(["user", "assistant"])` rejects any role value outside the two allowed, preventing arbitrary role injection into stored history |
| T-02.1-SC | Tampering | npm/Supabase CLI invocation | accept | No new package installed this plan; `supabase` CLI already verified present in RESEARCH.md Environment Availability -- no new install surface |
</threat_model>
