# Phase 02.1 — UI Design Contract

> Visual and interaction contract for: mobile responsiveness, persistent AgriMate AI chat, multi-listing bug fix, listing image bug fix, dark mode. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase does **not** introduce a new visual language. `components.json` shows shadcn already initialized (`base-nova` style, `neutral` base color, `lucide` icons) and `globals.css` already has complete light + dark CSS variable sets. The contract below is "apply the existing system correctly and consistently to the 6 named items," not a redesign. Full visual redesign is explicitly out of scope — that is Phase 4 per ROADMAP.md.

**Known existing inconsistency (do not propagate further in this phase):** `src/app/page.tsx` (landing) and several action buttons in dashboards (`owner/dashboard/page.tsx` line 104, `farmer/dashboard/page.tsx` line 115) use hardcoded `slate-950`/`emerald-*`/`amber-*` Tailwind colors instead of the shadcn `bg-primary`/`text-primary-foreground` tokens used everywhere else (browse page, cards, badges). This phase must **not** add new hardcoded color classes anywhere it touches — every new/modified element in this phase uses shadcn semantic tokens (`bg-primary`, `text-muted-foreground`, `bg-card`, `bg-destructive`, etc.) exclusively, specifically because dark mode (item 6) only works correctly through tokens — hardcoded `slate-950` literals do not flip with `.dark`. Fixing the landing page's existing hardcoded colors is out of scope for this phase (Phase 4 territory) unless it visibly breaks in dark mode; if it does, the minimum fix is swapping those literals for tokens, not a redesign.

## Spacing Scale

Declared values (must be multiples of 4) — matches what's already used throughout the codebase (`PageShell`, `Card`, dashboards):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps inside buttons/badges |
| sm | 8px | Compact element spacing (badge+button gap, nav link gap) |
| md | 16px | Default element spacing (`PageShell` mobile padding `p-4`, card gaps) |
| lg | 24px | Section padding (`PageShell` desktop padding `sm:p-6`, `space-y-6` between dashboard sections) |
| xl | 32px | Layout gaps (landing page section padding) |
| 2xl | 48px | Major section breaks (unchanged from existing landing page `py-20`-equivalent scale) |
| 3xl | 64px | Page-level spacing (landing hero `min-h-[80vh]` equivalent) |

**Exceptions for this phase:**
- Touch targets: every tappable control introduced or modified in this phase (nav links inside a mobile drawer, theme toggle button, hamburger trigger, chat send button) must be **minimum 44px × 44px** hit area on touch viewports, even where the visible icon is smaller (e.g. `ThemeToggle` button uses shadcn `Button size="icon"` which is `h-9 w-9` / 36px by default — bump to `size-11` / 44px specifically on screens `<sm:` via `h-11 w-11 sm:h-9 sm:w-9`, or accept `h-9 w-9` only if it sits inside a header row with adequate surrounding tap padding; prefer the explicit 44px floor for the hamburger trigger and theme toggle since they are new, isolated controls).
- Dashboard list rows (MOBILE-02 fix): switch `flex items-center justify-between` to `flex flex-col gap-2 sm:flex-row sm:items-center sm:justify-between` — introduces an 8px (`gap-2`) vertical gap between stacked title/meta and the badge/action row on narrow viewports, collapsing to the existing horizontal layout at `sm:` and above. No new spacing token needed beyond `sm` (8px) and `md` (16px) already declared.

## Typography

Matches the type scale already in active use across `PageShell`, dashboards, and cards — do not introduce new sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label / meta | 12px (`text-xs`) | 400 (regular) | 1.4 (existing `text-xs text-muted-foreground` pattern, e.g. badge captions, timestamps) |
| Heading | 20px (`text-xl`, `font-semibold tracking-tight`) | 600 (semibold) | 1.2 — matches `PageShell` `<h1>` exactly |

The AgriMate AI rebrand header (`CardTitle` in `ChatWidget`) stays at its existing `CardTitle` default size (shadcn default, ~18px/`text-lg`, weight 600) — do not change `CardTitle`'s own scale, only its text content and an added tagline line at 12px/`text-xs text-muted-foreground` weight 400.

## Color

Uses the existing shadcn neutral CSS-variable tokens already defined in `globals.css` for both `:root` and `.dark` — this phase's entire job for color is making sure **every new/modified element reads these variables**, not hex literals, so dark mode (THEME-01) works everywhere it's wired.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `bg-background` / `text-foreground` | Page background, `<body>`, main content area |
| Secondary (30%) | `bg-card`, `bg-muted`, `bg-secondary` | Cards (equipment list items, dashboard stat tiles, chat bubbles), nav header background (`border-b` on default background, no separate header fill), dropdown menus |
| Accent (10%) | `bg-primary` / `text-primary-foreground` | **Reserved for: primary CTA buttons only** — "Create listing" / "List equipment" button, "Send" button in chat, the active/pressed state of the theme toggle. Do not apply primary color to nav links, badges, or decorative icons. |
| Destructive | `bg-destructive` / `text-destructive-foreground` (existing `variant="destructive"` on `Badge`/`Button`) | Cancel booking, reject booking, delete listing confirmation — already in use, no change |

**Accent reserved for, explicit list:**
1. "Create listing" / "List equipment" primary button (currently hardcoded `bg-slate-950` in two empty-state CTAs — **this phase must convert these two specific buttons to `bg-primary text-primary-foreground` via the shadcn `Button` component** so they correctly invert in dark mode; this is in-scope because it is a direct dark-mode correctness bug, not a redesign)
2. Chat "Send" button (already correctly uses shadcn `<Button>` default variant = primary — no change needed)
3. Theme toggle's active/selected state indicator, if a 3-way Light/Dark/System control is used instead of a binary toggle (see Phase-Specific Component Contracts below)

**Dark mode swap (THEME-01):** No new colors — `.dark` block in `globals.css` already defines every token above. `attribute="class"` on `ThemeProvider` is mandatory (see RESEARCH.md Pitfall 1) because `@custom-variant dark (&:is(.dark *));` only reacts to a `.dark` class, not `data-theme`.

## Phase-Specific Component Contracts

The template's generic sections above cover tokens; this section pins down the concrete per-item visual decisions RESEARCH.md flagged as "Claude's Discretion."

### 1. Mobile nav header collapse (MOBILE-01)

**Decision:** Use shadcn `Sheet` (slide-in drawer from the right) triggered by a hamburger icon button, replacing the flat `<nav className="flex gap-3">` row on viewports below `sm:` (640px). At `sm:` and above, render the existing horizontal `<nav>` unchanged.

- Trigger: `Button variant="ghost" size="icon"` with `Menu` icon (lucide-react), visible only `<sm:` (`className="sm:hidden"`), minimum 44×44px tap target.
- Existing horizontal nav: add `hidden sm:flex` (currently always renders — must be hidden below `sm:` once the Sheet trigger takes over).
- `Sheet` content: `side="right"`, width `w-64`, contains the same nav `<Link>` list stacked vertically (`flex flex-col gap-1`) plus `NotificationBell` and `LogoutButton` at the bottom, separated by a `border-t pt-3` divider — do not duplicate the bell/logout outside the header on mobile; move them inside the Sheet so the header row stays to just: brand label + hamburger trigger.
- Header row at `<sm:` becomes: `<span className="text-sm font-medium">AgriRent — Owner</span>` (unchanged) + hamburger trigger only. `NotificationBell` and `LogoutButton` relocate into the Sheet on mobile; remain inline in the header at `sm:` and above (current behavior unchanged at desktop widths).
- This pattern applies identically to both `(owner)/layout.tsx` and `(farmer)/layout.tsx` — same `Sheet` structure, just the farmer one has one extra nav link (`Favorites`).

### 2. Dashboard list row wrapping (MOBILE-02)

**Decision:** Both `owner/dashboard/page.tsx`'s "All Bookings" list and `farmer/dashboard/page.tsx`'s "My Bookings" list change their `<li>` className from:
```
flex items-center justify-between rounded-md border p-2 text-sm
```
to:
```
flex flex-col gap-2 rounded-md border p-2 text-sm sm:flex-row sm:items-center sm:justify-between
```
No copy or data changes — purely a flex-direction/gap fix so the title+meta block and the badge+action-button block stack vertically under ~640px instead of compressing into one cramped row. The inner `<div className="flex items-center gap-2">` (badge + action buttons) stays as-is but should get `flex-wrap gap-2` added defensively in case 2-3 action buttons plus a badge still overflow a narrow stacked row.

### 3. AgriMate AI chat rebrand — file-by-file scope (CHAT-BRAND-01)

Apply "🥇 AgriMate AI" / tagline exactly where the **name** renders as a UI label. Confirmed locations from RESEARCH.md + this session's grep (`grep -rln "AgriRent Assistant"` → 2 files):

| `src/components/chat/chat-widget.tsx` line 102 (empty-state body line) | `"Ask a question about how booking and rental works on AgriRent."` | `"Get instant answers about booking, pricing, or how rentals work on AgriRent."` (per Copywriting Contract above) |

### 4. Owner dashboard equipment thumbnail (BUG-IMAGE-01 visual contract)

**Decision:** Mirror `browse/page.tsx` lines 47-66 exactly, adapted to the dashboard's `<ul>`/`<li>` list layout (not a card grid) since the "My Equipment" section is a compact list, not the browse grid.

- Each `<li>` becomes a `flex items-center gap-3` row (was a single-line text link): a fixed-size thumbnail + the existing title/rate text + (optionally) the existing edit affordance.
- Thumbnail container: `relative h-14 w-14 shrink-0 overflow-hidden rounded-md bg-muted` (56×56px square — deliberately smaller than browse's `aspect-video` card image since this is a dense list row, not a card grid).
- Image: `next/image` with `fill` + `className="object-cover"`, `src={listingService.getEquipmentImageUrl(firstImage.storage_path)}`, `alt={item.title}` — identical call pattern to browse page, just different container dimensions.
- Fallback (no image): same pattern as browse — `<div className="flex h-full items-center justify-center text-xs text-muted-foreground">No photo</div>` inside the same 56×56 container, not a different fallback string or icon.
- Layout: `<li className="flex items-center gap-3 rounded-md border p-2 text-sm">` containing the thumbnail div, then a `<div className="min-w-0 flex-1">` wrapping the existing `<Link href=".../edit">{item.title} — ₹{item.rate}/{item.rate_unit}</Link>` (add `truncate` to the title span if it's a long title, since the row is now visually denser).

### 5. Multi-listing bug fix — post-create confirmation state (BUG-LISTING-01 visual contract)

**Decision:** No new design needed beyond the existing `toast.success(result.message)` (sonner) already called before `router.push(...)` in `equipment/new/page.tsx` line 91 — confirmed this already fires correctly today; only the redirect *target* on line 92 changes from `"/browse"` to `"/owner/dashboard"`. This matches the working pattern in `EditEquipmentForm.tsx` line 82 exactly. No new toast copy, no new confirmation screen — the existing success toast + landing on a dashboard that now visibly shows the new listing (thanks to the BUG-IMAGE-01 fix above) **is** the confirmation state. Do not build a separate "listing created" interstitial page — that would be scope creep beyond the one-line root-cause fix RESEARCH.md identified.

### 6. Dark mode toggle (THEME-01)

**Decision:** Binary Sun/Moon icon-button toggle (not a 3-way Light/Dark/System dropdown) — matches the simplest correct pattern already half-proven via `sonner.tsx`'s `useTheme()` call, and keeps the control to a single tap rather than a menu, which is friendlier on the mobile-responsive surfaces this same phase is fixing.

- Component: new `src/components/theme/theme-toggle.tsx`, exact pattern from RESEARCH.md Pattern 2 (`Button variant="ghost" size="icon"`, `Sun`/`Moon` from lucide-react, `hidden dark:block` / `block dark:hidden` swap).
- Placement: inside both `(owner)/layout.tsx` and `(farmer)/layout.tsx` header's right-hand `<div className="flex items-center gap-2">`, positioned **before** `NotificationBell` (left-to-right order: ThemeToggle, NotificationBell, LogoutButton) — so it reads as a settings-adjacent control, not mixed into notification/account actions.
- On mobile (`<sm:`), per the MOBILE-01 Sheet decision above, `ThemeToggle` moves inside the `Sheet` drawer alongside `NotificationBell`/`LogoutButton` — same relative order (Theme, Notifications, Logout) at the bottom of the drawer's link list.
- Public/landing nav (`src/app/page.tsx`): the landing page currently has no header/nav bar at all (it's a single scrolling hero page with inline `Get started`/`Log in` links, not a persistent header). **Decision: do not add a persistent header just to host a theme toggle on the landing page this phase** — that's a structural change beyond the 6 named items. If a quick toggle is wanted pre-login, the simplest in-scope option is omitting it from the landing page entirely for now (default to `system` theme there, same as everywhere) and only exposing the manual toggle inside the authenticated farmer/owner layouts where a header already exists. Flag this as a known gap, not a silent omission.
- Default theme: **`system`** (`defaultTheme="system" enableSystem`), per RESEARCH.md Open Question 2 recommendation — zero extra code, matches Shadcn's own documented default, and no stated user preference exists to override it.
- `ThemeProvider` wraps `{children}` in `src/app/layout.tsx` with `attribute="class"` and `disableTransitionOnChange`; `<html>` gets `suppressHydrationWarning` (RESEARCH.md Pattern 1 / Pitfall 2) — both are non-negotiable for correctness, not a design choice.
