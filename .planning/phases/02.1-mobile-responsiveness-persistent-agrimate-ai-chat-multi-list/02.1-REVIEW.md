# Phase 02.1: Code Review Report

**Reviewed:** 2026-06-27T00:00:00Z
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

This phase adds mobile responsiveness (Sheet-based nav drawer, theme toggle) and persistent AgriMate AI chat (Supabase-backed history, streaming chat widget). The chat persistence layer (migration, service, action, schema, tests) is solid: RLS policies correctly scope to `auth.uid()`, the service layer never trusts client input, and the Zod schema enforces a sane length cap. The mobile-nav Sheet drawer is functionally wired into both layouts but has a real UX-breaking defect: nav links inside the drawer never close it, so the mobile menu stays open across in-app navigations. There is also one accessibility regression (missing `SheetTitle`) and one already-flagged-but-still-real maintainability issue inherited into this phase's diff (`types/database.ts` checked in as UTF-16LE, making the file unreadable to standard text tooling and breaking diff review for every future schema change). One genuinely dead Server Action (`getChatHistoryAction`) ships unused.

## Critical Issues

### CR-01: Mobile nav Sheet drawer never closes on navigation, leaving stale open state across routes

**File:** `src/app/(farmer)/layout.tsx:69-103`, `src/app/(owner)/layout.tsx:63-91`

**Issue:** Both `<Sheet>` instances are fully uncontrolled (no `open`/`onOpenChange` prop, no local state). The `<Link>` elements rendered inside `<SheetContent>` (lines 84-95 farmer, 77-83 owner) have no `onClick` handler that closes the sheet. Because `(farmer)/layout.tsx` and `(owner)/layout.tsx` wrap every route in their respective groups and are NOT remounted by the Next.js App Router on client-side navigation between sibling routes (e.g. `/farmer/dashboard` → `/farmer/chat`), the `Sheet`'s internal open-state persists across the navigation. A farmer/owner on mobile who opens the drawer, taps "Chat", and lands on the chat page will find the drawer still rendered open over the new page (or, depending on base-ui's exact unmount timing relative to the route transition, the overlay/backdrop staying mounted and blocking input). This is a shipped, user-visible breakage of the core navigation flow this phase exists to build — not a cosmetic issue.

**Fix:** Either wrap the link click handler to close the sheet, or use `SheetClose` to render the nav links so base-ui closes the popup automatically on activation:
```tsx
import { Sheet, SheetClose, SheetContent, SheetTrigger } from "@/components/ui/sheet";

<SheetContent side="right" className="w-64">
  <nav className="flex flex-col gap-1 p-4">
    <SheetClose render={<Link href="/farmer/dashboard" className="text-muted-foreground hover:text-foreground" />}>
      Dashboard
    </SheetClose>
    {/* ...repeat for each Link... */}
  </nav>
  ...
</SheetContent>
```
Or, if `SheetClose` cannot wrap `Link` cleanly, lift `open` into local state in a small client component and call `setOpen(false)` in each link's `onClick`.

## Warnings

### WR-01: `types/database.ts` checked in as UTF-16LE, breaking text tooling and diff review

**File:** `types/database.ts:1-528`

**Issue:** The file is UTF-16LE-encoded with CRLF line endings (confirmed via `file types/database.ts`), unlike every other `.ts` file in the repo. This was knowingly regenerated and committed in this phase's diff (commit `c2abaef`, "regenerate database types after chat_messages migration push") rather than being pre-existing baseline drift. Consequences: `git diff` shows this file as a binary blob (`Bin 28734 -> 30302 bytes`) with zero line-level review possible — a schema-affecting file is effectively un-reviewable in PRs going forward — and any tool/script that greps or text-processes this file (CI lint scripts, this very review's tooling) silently fails to match patterns inside it. While TypeScript's compiler tolerates the BOM/encoding, this is a real, growing maintainability cost that compounds every time the schema changes.

**Fix:** Regenerate with explicit UTF-8 output instead of relying on PowerShell's `>` default:
```powershell
npx supabase gen types typescript --linked | Set-Content -Encoding utf8 types/database.ts
```
Re-commit once as a pure encoding fix (no content change) so future schema diffs are reviewable.

### WR-02: Dead Server Action `getChatHistoryAction` is exported but never called

**File:** `src/app/actions/chat.actions.ts:46-55`

**Issue:** `getChatHistoryAction` is defined and exported, but no caller exists anywhere in `src/`. Both `farmer/chat/page.tsx` and `owner/chat/page.tsx` call `chatService.getChatHistory(userData.user.id)` directly instead of going through this action (which itself just re-derives the user and forwards to the same service function). This is unused code shipped alongside the feature it was presumably built for.

**Fix:** Either delete `getChatHistoryAction` if the direct service call is the intended pattern (Server Components calling services directly is consistent with the rest of the codebase), or, if it's meant for a future client-side refresh use case, leave a comment explaining why it's intentionally unused for now. Given no client component currently needs to re-fetch history without a full page load, deleting it is the simpler fix.

### WR-03: `chat-widget.tsx` ignores `saveChatMessageAction` failures for both user and assistant turns

**File:** `src/components/chat/chat-widget.tsx:47`, `src/components/chat/chat-widget.tsx:89`

**Issue:** Both calls to `saveChatMessageAction(...)` are awaited but their `ActionResult` is discarded (`await saveChatMessageAction("user", content);` with no `.success` check). If the insert fails (e.g., RLS denial, transient DB error, content exceeding the 2000-char Zod cap because a user pastes a huge message), the user sees no indication that their message wasn't persisted — the in-memory UI state shows it, but `farmer/chat` or `owner/chat` will silently lose it on next reload. This conflicts with the project's own `{success, message, data}` convention existing precisely so callers can react to failure.

**Fix:** At minimum, log/console.error and optionally toast on failure for the assistant save (the most damaging case — it represents the AI's costly response):
```ts
const saveResult = await saveChatMessageAction("assistant", accumulatedContent);
if (!saveResult.success) {
  console.error("chat-widget: failed to persist assistant reply", saveResult.message);
}
```

### WR-04: `chat-widget.tsx` uses array index as React list key for a mutating, streaming message list

**File:** `src/components/chat/chat-widget.tsx:118-129`

**Issue:** `messages.map((message, index) => <div key={index} ...>` uses the array index as the key. The list is append-only here today, so this happens to be stable in practice, but it's a known anti-pattern specifically because the content of the last element mutates on every streamed token (`updated[updated.length - 1] = {...}` at line 81). React's reconciliation for keyed lists assumes key identity maps to a stable logical item; index-based keys silently break if any future change inserts/removes/reorders messages (e.g., adding message deletion, optimistic-message rollback on `saveChatMessageAction` failure per WR-03, or retry-on-error UI). Combined with WR-03, a natural fix for one (rolling back a failed optimistic message) would introduce real bugs under the other (index-keyed list rerender producing flicker/incorrect content swap).

**Fix:** Generate a stable id per message at creation time instead of relying on array position:
```ts
type Message = { id: string; role: "user" | "assistant"; content: string };
// ...
const nextMessages: Message[] = [...messages, { id: crypto.randomUUID(), role: "user", content }];
```

### WR-05: `buildSystemPrompt()` fetches the entire unfiltered equipment table on every chat turn with no caching, despite the file's own comment claiming otherwise

**File:** `src/lib/services/ai.service.ts:60-101`

**Issue:** The doc comment on `buildSystemPrompt` (lines 60-64) explicitly states "data is cached per request — not per user session," implying there is request-scoped caching. There is none — `getChatCompletion` calls `buildSystemPrompt()` unconditionally on every invocation (line 120), which calls `listingService.getAllEquipment()` with no memoization, cache, or request-scoping mechanism. This isn't flagged for the performance cost (out of scope), but the comment is actively misleading about the code's actual behavior, which will cause a future maintainer to incorrectly assume caching exists when reasoning about staleness or load.

**Fix:** Either implement the caching the comment describes (e.g., `React.cache()` per request, or a short TTL) or correct the comment to state plainly that it refetches every turn:
```ts
/**
 * Builds a dynamic system prompt with live marketplace context. Called
 * once per /api/chat request — no caching; every turn re-fetches the
 * full active listing set from listingService.getAllEquipment().
 */
```

### WR-06: `SheetContent` renders without a `SheetTitle`/`SheetDescription` in both mobile nav drawers, dropping required dialog accessibility labelling

**File:** `src/app/(farmer)/layout.tsx:82-103`, `src/app/(owner)/layout.tsx:76-91`

**Issue:** `base-ui`'s `Dialog.Popup` (which `SheetContent` wraps, per `src/components/ui/sheet.tsx:52`) is a dialog-role element expected to be labelled via `aria-labelledby`/`aria-describedby` pointing at a `Title`/`Description`. Neither `SheetTitle` nor `SheetDescription` is rendered in either nav drawer's `SheetContent`. Screen reader users opening the mobile menu get an unlabelled dialog region, and base-ui (like Radix) typically emits a console warning in development for a missing accessible title.

**Fix:** Add a visually-hidden title (Shadcn's own docs pattern for icon-only/menu sheets):
```tsx
<SheetContent side="right" className="w-64">
  <SheetTitle className="sr-only">Navigation menu</SheetTitle>
  <nav className="flex flex-col gap-1 p-4">
    ...
```

## Info

### IN-01: `EquipmentFormValues` widens `rate` to `string | number` via an unchecked cast at the render call site

**File:** `src/app/(owner)/equipment/new/page.tsx:182`

**Issue:** `value={field.value as string | number}` casts away type information rather than letting the input's controlled value type be inferred normally. This isn't incorrect today (the comment above the type alias at lines 40-46 explains the `z.input` vs `z.infer` mismatch reasoning), but the `as` cast at the JSX call site suppresses any future type-checking if the schema's input type changes shape.

**Fix:** Not blocking; consider deriving a narrower prop type once, e.g. a small `NumericFieldValue = string | number` alias used both at the type-alias level and at this call site, so the cast documents intent in one place instead of being repeated inline.

### IN-02: Duplicated Sheet-drawer nav markup between `(farmer)/layout.tsx` and `(owner)/layout.tsx`

**File:** `src/app/(farmer)/layout.tsx:69-103`, `src/app/(owner)/layout.tsx:63-91`

**Issue:** The entire `<Sheet>`/`<SheetTrigger>`/`<SheetContent>` block (including the `ThemeToggle`/`NotificationBell`/`LogoutButton` footer) is duplicated near-verbatim between the two layouts, differing only in the nav links list. This duplicates the WR-01/WR-06 fixes across two files when applied.

**Fix:** Extract a shared `<MobileNav links={...} userId={...} />` client component used by both layouts, so the close-on-navigate fix (CR-01) and the `SheetTitle` fix (WR-06) only need to be made once.

### IN-03: `chat.actions.ts`'s `ActionResult.data: unknown` discards the strongly-typed `ChatMessageRow`/`ChatMessageRow[]` shape from the service layer

**File:** `src/app/actions/chat.actions.ts:7-11`

**Issue:** `ActionResult` types `data` as `unknown`, even though `chatService.saveChatMessage`/`getChatHistory` return precisely-typed `ServiceResult<ChatMessageRow>` / `ServiceResult<ChatMessageRow[]>`. Both actions just pass the service result straight through (lines 36-40, 54), so the type narrowing is thrown away for no functional reason — callers that do inspect `.data` (none currently do, per WR-02/WR-03) would have to re-cast.

**Fix:** Make `ActionResult` generic, matching the service layer's own pattern:
```ts
type ActionResult<T = unknown> = { success: boolean; message: string; data: T | null };
export async function saveChatMessageAction(...): Promise<ActionResult<Tables<"chat_messages">>> { ... }
```

### IN-04: `chat.service.test.ts` mock chain assumes a fixed `select().eq().order()` call shape, coupling tests to implementation rather than behavior

**File:** `src/lib/services/chat.service.test.ts:19-31`

**Issue:** `makeFakeSupabaseHistory` hand-builds a mock that only supports exactly one `.select().eq().order()` chain. This is a reasonable, common Supabase-mock pattern and not a functional bug, but it means any future change to `getChatHistory`'s query (e.g., adding `.limit()` or a second `.eq()` for soft-delete filtering) will require updating the mock shape in lockstep rather than the test failing for a behavioral reason. Noted for awareness, not blocking — this matches the existing `auth.service.test.ts` precedent the author cites.

**Fix:** No action required now; if `getChatHistory`'s query grows more complex, consider a more generic chainable mock builder shared across service tests.
