# Phase 03.5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase has no CONTEXT.md or RESEARCH.md (urgent insertion; `/gsd-discuss-phase` and `/gsd-plan-phase` not yet run — same situation as Phase 03.4). This is a **content/copy refresh of one existing page** (`src/app/page.tsx`), not new functionality, no new routes, no new components beyond what already exists. All design tokens (spacing, typography, color) are reused from the already-approved Phase 03.1 nav contract and Phase 03.4's profile contract — no new tokens introduced. This UI-SPEC is therefore primarily a **section-by-section content contract**: what each section says, in what order, using which existing primitives, not a new visual system.

The current `src/app/page.tsx` (read in full) has 4 sections: hero, "everything you need" feature grid (6 items), a CTA banner, and a footer. It predates: reviews/ratings, the realtime notification bell, branded transactional emails, the AgriMate AI brand mark, and the richer role-based dashboards. The feature grid's copy is generic/aspirational ("Get recommendations and answers instantly") rather than reflecting what's actually shipped today.

## Spacing Scale

Declared values (must be multiples of 4) — reused from Phase 03.1/03.4's established scale, no new values introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (badge dot, inline icon spacing — matches existing hero badge pattern) |
| sm | 8px | Tight gaps within a feature card (icon block margin-bottom is currently `mb-3`/12px — keep as-is, not a new token) |
| md | 16px | Default gaps between grid items already in use (`gap-6` on the feature grid is 24px — see `lg` below; `gap-4`/`gap-3` used for CTA button rows) |
| lg | 24px | Feature card internal padding (`p-5`, close enough to existing convention — keep current `p-5`/20px as an existing exception, do not force to exactly 24px), feature grid `gap-6` |
| xl | 32px | Spacing between hero copy elements (`space-y-6`) |
| 2xl | 48px | Section-to-section rhythm if a new section is added (matches existing `py-20` section padding is actually 80px — see exception below) |
| 3xl | 64px | Not newly introduced; existing CTA banner uses `py-16` (64px) |

Exceptions (already present in the current page, kept as-is — not introduced by this phase):
- Feature grid section uses `py-20` (80px vertical padding) — pre-existing, off the declared 4-step scale's listed tokens but already a multiple of 4; do not change.
- Feature card padding is `p-5` (20px) — pre-existing, multiple of 4, keep consistent across all 6 (or more) feature cards; if a new section is added, match this `p-5` convention rather than inventing a new card padding value.
- Hero icon-badge "dot" uses `h-2 w-2` (8px) — pre-existing, unchanged.

## Typography

Reuses the project's established type scale already visible in the current `page.tsx` — no new size or weight introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (H1, hero headline) | 36px mobile → 60px desktop (`text-4xl sm:text-5xl lg:text-6xl`) | 700 (`font-bold`) | 1.2 (`tracking-tight`) |
| Heading (H2, section titles) | 24px → 30px (`text-2xl sm:text-3xl`) | 600 (`font-semibold`) | 1.2 (`tracking-tight`) |
| Body (paragraph copy, feature descriptions) | 16px (`text-sm`/`text-base` mixed — hero subhead is `text-base sm:text-lg`, feature card body is `text-sm`) | 400 (regular) | 1.5 (`leading-6`/`leading-7`) |
| Label (feature card title, badge text) | 14-16px (`font-medium` feature titles use base size; hero badge uses `text-sm`) | 500 (`font-medium`) | 1.5 |

Total distinct sizes in use: 4 (matches the declared 3-4 budget) — 14px/16px/24-30px/36-60px responsive clusters. Total weights: 2 (400 regular, 600/700 for headings — `font-medium` 500 on feature titles is the one pre-existing deviation from a strict 2-weight rule; kept as-is since it's already shipped and not part of this phase's content-only scope — do not introduce a 3rd new weight beyond this pre-existing one).

Never used for primary CTA button fills (those stay `bg-slate-950`/dark per existing convention) and never used as a blanket link/hover color across the whole page.

## Section-by-Section Content Contract

This is the actual deliverable for this phase — what changes in `src/app/page.tsx`.

### 1. Hero section (KEEP structure, copy unchanged — already accurate)

No changes needed. "Rent farm equipment without the friction." + the farmer/owner two-tone split card already accurately describes the core value prop. The `/globe.svg` placeholder icon in the hero panel header may optionally be replaced with `AgriMateAIIcon` (variant `navbar`) IF the executor wants a branded touch here — **not required**, this is Claude's Discretion, not a locked decision, since the hero panel's icon currently represents "AgriRent" generically, not the AI assistant specifically. Default: leave as-is unless executor judges the swap clearly improves brand cohesion without misrepresenting the icon's meaning (the hero panel isn't about AI — don't force the AI mark where it doesn't belong).

### 2. Feature grid section — REWRITE required (currently stale/generic)

Replace the 6 `features` array entries. Required content additions per the phase's scope (equipment browsing/booking, owner management, full lifecycle, reviews, realtime notifications, role dashboards, branded emails, AI chat, auth) — condense into exactly **6 cards** (matches existing 3-column grid layout, `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3`, do not expand the grid to 7+ which breaks the 3-column rhythm without also changing layout, which is out of scope):

| # | Icon (lucide) | Title | Description |
|---|---|---|---|
| 1 | `Search` | Browse & book | Search tractors, harvesters, and more by category or location, then request a date range — pricing is calculated automatically. |
| 2 | `ListCheck` | Owners stay in control | List equipment in minutes and approve or reject every booking request from a dedicated dashboard. |
| 3 | `ShieldCheck` | No double-booking | Availability is enforced at the database level on every request — never just a UI check. |
| 4 | `Star` | Real reviews | Farmers rate and review equipment after every completed rental, visible on the listing page. |
| 5 | `Bell` | Live notifications | A real-time notification bell and branded email alerts keep both sides updated the moment a booking status changes. |
| 6 | `MessageSquareText` (or `AgriMateAIIcon` inline, see note) | AgriMate AI | Ask AgriMate AI rental questions any time — booking policies, equipment tips, and more, answered instantly. |

Icon import change required: swap `Trees` out (no longer represents a distinct shipped feature once reviews/notifications/AI are added) for `Bell` (new import from `lucide-react`, already the icon family in use elsewhere e.g. `NotificationBell.tsx`). Card #6 may use the actual `AgriMateAIIcon` component instead of a generic lucide icon inside the same icon-chip treatment (`h-10 w-10 rounded-lg bg-emerald-100` wrapper) — **recommended**, since AgriMate AI now has a real brand mark and this is the one card where using it is unambiguously correct (unlike the hero panel). If used, render `<AgriMateAIIcon variant="navbar" className="h-5 w-5" />` inside the existing icon-chip wrapper div, replacing the `<feature.icon className="h-5 w-5" />` pattern for that one card only (executor's call whether to special-case the render or keep the array driving a uniform lucide-only render — either satisfies this contract; using the real mark is the only meaningfully better option, not a hard requirement).

Section heading copy: keep "Everything you need" / "From discovery to booking to feedback — all in one place." — still accurate, no change required.

### 3. NEW section (recommended, not mandatory) — "Built for both sides" or "How it works"

**Claude's Discretion** — the existing hero panel already does a farmer/owner split, and the feature grid covers functionality. Adding a third "how it works" step-by-step (1. Browse → 2. Request → 3. Get approved → 4. Review) is optional polish, not required by the phase goal ("copy accurately represents shipped functionality" — already satisfied by an accurate feature grid alone). If the executor adds this section: reuse the feature-card visual pattern (`rounded-xl border bg-white p-5`) in a 4-column or numbered-list layout, keep it between the feature grid and the CTA banner, and do not introduce a new color or icon style for it.

### 4. CTA banner section (KEEP structure, minor copy check)

"Ready to get started?" / "Join AgriRent today and start renting or listing farm equipment in your area." — still accurate, no change required. Buttons "Create free account" / "Log in" unchanged.

### 5. Footer (KEEP, verify accuracy)

"AgriRent — farm equipment rental marketplace" — still accurate. No change required unless executor wants to add the live deployment domain reference (optional, not required): e.g. could stay exactly as-is. **Do not** add unimplemented links (no footer nav links exist today — do not imply a sitemap that doesn't exist).
