## Warnings

### WR-01: String-equality branch on a translatable/editable display title is a fragile coupling between copy and icon logic

**File:** `src/app/page.tsx:153`
**Issue:** The icon-swap logic branches on `feature.title === "AgriMate AI"` to decide whether to render the brand mark `AgriMateAIIcon` or the generic `feature.icon`. This ties rendering logic to a copywriting string with no compile-time or runtime safety net. If a future content edit renames the card title (e.g., to "AgriMate AI Assistant" or a localized string), the conditional silently stops matching, the card silently falls back to rendering `feature.icon` (which is the placeholder `MessageSquareText`, a generic chat-bubble icon), and nothing errors or warns — the regression would only be caught by a human visually inspecting the page, which is exactly the failure mode that produced the stale copy this phase was created to fix in the first place.
**Fix:** Key the branch on a stable, non-copy identifier instead of the display string, e.g. add an explicit discriminant field to the array entry:
```tsx
const features = [
  // ...
  {
    id: "agrimate-ai",
    icon: MessageSquareText,
    title: "AgriMate AI",
    description: "...",
  },
];
// ...
{feature.id === "agrimate-ai" ? (
  <AgriMateAIIcon variant="navbar" className="h-5 w-5" />
) : (
  <feature.icon className="h-5 w-5" />
)}
```

### WR-02: Pre-existing unvalidated interpolation of the `code` search param into a redirect URL (not introduced by this phase, but live in the reviewed file)

**File:** `src/app/page.tsx:49-53`
**Issue:** `code` comes straight from `searchParams` (fully attacker-controlled query string) and is interpolated unescaped into a relative redirect target: `redirect(`/auth/callback?code=${code}`)`. Because the target is a same-origin relative path, this is not a classic open redirect to an external host, but an attacker can still inject extra query syntax (e.g. visiting `/?code=abc&role=owner` makes `code` literally equal `abc` with `role=owner` parsed as a second top-level param by Next's `searchParams`, which is benign here — but if `code` itself contains a value like `abc#fragment` or `abc&role=owner` as a single string value, e.g. via `/?code=abc%26role%3Downer`, the decoded value becomes `abc&role=owner` and the resulting redirect string becomes `/auth/callback?code=abc&role=owner`), letting the caller smuggle an arbitrary `role` (or other) parameter into `/auth/callback`'s query string that the route handler does parse and trust (`searchParams.get("role")` on `auth/callback/route.ts:41`, used to upsert a new user's role in `oauthRoleSchema.safeParse(roleParam)`). This is only exploitable for a *new* user's initial role assignment (existing users already have a `profile.role` from the DB and take the early-return branch), so the blast radius is "a first-time OAuth signup can choose owner vs farmer for themselves" — which the signup flow already lets them choose anyway via the legitimate `role` param — but it's still unvalidated string concatenation building a URL from raw user input, which is exactly the pattern CLAUDE.md's threat-modeling conventions ask to avoid trusting from the client.
**Fix:** Use `URLSearchParams` to build the redirect target instead of manual string interpolation, and/or validate that `code` matches the expected shape (Supabase auth codes are UUIDs) before forwarding it:
```tsx
if (code) {
  const params = new URLSearchParams({ code });
  redirect(`/auth/callback?${params.toString()}`);
}
```
This doesn't fully close the "new user picks their own role" question (that's a product decision, not a bug), but it removes the ability to inject additional query keys via a crafted `code` value. Since this code was not modified by the 03.5 diff, treat this as a backlog item rather than a blocker for this phase — but it should not be re-introduced or copied as a pattern elsewhere.

## Info

### IN-01: `icon: MessageSquareText` retained as a documented-but-undocumented dead value in the array literal

**File:** `src/app/page.tsx:39-43`
**Issue:** The "AgriMate AI" entry keeps `icon: MessageSquareText` purely so the array's inferred type stays uniform across all 6 entries — per the plan and SUMMARY.md, this value is intentionally never rendered (the JSX always branches away from it via `feature.title === "AgriMate AI"`). That rationale exists in the plan and SUMMARY.md but **not** as a comment in the source file itself. Anyone reading `page.tsx` in isolation (without the planning artifacts) will reasonably assume this icon is live and either render-test it or "fix" it by removing the now-apparently-unused `MessageSquareText` import, only to discover a TypeScript shape mismatch.
**Fix:** Add a one-line inline comment at the array entry to self-document the intent:
```tsx
{
  // icon is unused for this card — AgriMate AI brand mark is rendered instead (see JSX below)
  icon: MessageSquareText,
  title: "AgriMate AI",
  description: "...",
},
```

### IN-02: Hardcoded `/globe.svg` placeholder image continues to be used as the hero panel's brand icon

**File:** `src/app/page.tsx:95-102`
**Issue:** The hero panel renders `next/image`'s default Next.js starter asset `/globe.svg` (a generic globe icon) as the "AgriRent" brand mark, labeled with alt text `"AgriRent"`. This is unrelated to the actual `AgriMateAIIcon` brand asset that now exists in the codebase and is used elsewhere on this same page (feature card 6) and in the navbar. The UI-SPEC explicitly marked this as "optional/Claude's discretion, not required" for this phase, so it is correctly out of scope for this diff — flagging only as a forward-looking inconsistency: the page now uses two different "brand" visuals (a generic globe SVG in the hero, the real AgriMate leaf mark in the feature grid) which undercuts the polish this phase was trying to add.
**Fix:** Out of scope for this phase; consider swapping `/globe.svg` for a proper AgriRent logo/wordmark in a future polish pass (e.g., Phase 04 UI design polish, which the SUMMARY.md already lists as a downstream-affected phase).

### IN-03: `key={feature.title}` uses the same display string this card now also uses for icon-branching logic

**File:** `src/app/page.tsx:149`
**Issue:** Not a bug today (titles are unique and stable within a single render), but combined with WR-01, the title string is now doing double duty as both a React list key and a render-logic discriminant. This is a minor coupling smell — if a future edit introduces two cards with the same title (e.g., A/B copy testing) or relies on `feature.title` for a third purpose, the failure modes compound.
**Fix:** If the `id` field suggested in WR-01 is added, use it as the `key` too: `key={feature.id}`. Otherwise no action needed at present.
