## Root Cause — Forced-Light Page Token Bleed (drives Sections "Forced-Light Override" and "Component Token Audit" below)

**[LOCKED — user]** Phase 2.1 fixed only the page *background* (removed `/60` alpha from `bg-gradient-to-b from-emerald-50/60 to-white`). It did not fix the auth card's interior, because these elements read color from CSS custom properties that flip when an ancestor `<html class="dark">` is present, while the card itself is hardcoded `bg-white`:

| Element | Token(s) it inherits | Light value (`:root`) | Dark value (`.dark`) | Effect when `.dark` is ancestor + card is `bg-white` |
|---|---|---|---|---|
| `FormLabel` / `Label` (no explicit color class) | `--foreground` (via Tailwind default text color, body inherits) | `oklch(0.145 0 0)` (near-black) | `oklch(0.985 0 0)` (near-white) | Label text turns near-white on white card → invisible |
| `Input` border | `--input` (via `border-input`) | `oklch(0.922 0 0)` (light gray) | `oklch(1 0 0 / 15%)` (15% white) | Border barely visible on white card |
| `Input` placeholder | `--muted-foreground` (via `placeholder:text-muted-foreground`) | `oklch(0.556 0 0)` (mid gray) | `oklch(0.708 0 0)` (lighter gray) | Placeholder washes out, low contrast on white |
| "OR" divider line (`<span className="w-full border-t" />`) | `--border` (Tailwind default border color) | `oklch(0.922 0 0)` | `oklch(1 0 0 / 10%)` | Divider line nearly invisible |
| `RadioGroupItem` border/bg | `--input`, `--primary` | light values | dark values (inverted: `--primary` becomes near-white in dark) | Unselected radio border faint; selected radio can render wrong contrast |
| Google button border (`border-slate-300` literal — already hardcoded, NOT a var) | n/a — hardcoded Tailwind color | `slate-300` always | `slate-300` always | **Not actually a CSS-var problem** — this one was already a literal class on the call site. (See note below.) |

**Important correction to carry into planning:** the Google OAuth button's border is set via a literal `border-slate-300` class directly in `google-oauth-button.tsx` (not a CSS var), so it does NOT bleed from `.dark` — it should already render at `slate-300` regardless of theme. If the user is still observing an invisible Google border, the more likely cause is the *card surrounding it* — `bg-white` next to a `border-slate-300` border is low-but-nonzero contrast (≈1.3:1), which can look "invisible" even in light mode at a glance, independent of the dark-mode bug. The systemic fix below does not need to touch this token (it's already hardcoded correctly), but the **redesign** pass should deliberately increase this border's contrast (see Color section) so it is unambiguously visible standalone.

### Required Fix — `.force-light` override class

**[LOCKED — user]** Declare a single reusable class in `src/app/globals.css`, scoped narrowly, that re-declares every token consumed by these pages' components back to their `:root` values — winning over an ancestor `.dark` by CSS custom-property cascade (a more-specific/later rule on a wrapper element overrides the inherited value regardless of `.dark` on `<html>`):

```css
/* Forces light-mode token values on pages that must never respect the
   user's dark-mode preference (login, signup, forgot-password,
   reset-password, home). Apply to each page's root wrapper element.
   Re-declares every token these pages' components actually consume —
   keep this list in sync if a new shadcn primitive is added to these
   pages. Replaces the narrow border-slate-300 per-input patches from
   the 260627-ph2 quick task; remove those once this lands. */
.force-light {
  --background: oklch(1 0 0);
  --foreground: oklch(0.145 0 0);
  --card: oklch(1 0 0);
  --card-foreground: oklch(0.145 0 0);
  --popover: oklch(1 0 0);
  --popover-foreground: oklch(0.145 0 0);
  --primary: oklch(0.205 0 0);
  --primary-foreground: oklch(0.985 0 0);
  --secondary: oklch(0.97 0 0);
  --secondary-foreground: oklch(0.205 0 0);
  --muted: oklch(0.97 0 0);
  --muted-foreground: oklch(0.556 0 0);
  --accent: oklch(0.97 0 0);
  --accent-foreground: oklch(0.205 0 0);
  --destructive: oklch(0.577 0.245 27.325);
  --border: oklch(0.922 0 0);
  --input: oklch(0.922 0 0);
  --ring: oklch(0.708 0 0);
}
```

**Application:** add `force-light` to the outermost `<main>` wrapper class list on each of: `src/app/(auth)/login/page.tsx`, `src/app/(auth)/signup/page.tsx`, `src/app/(auth)/forgot-password/page.tsx`, `src/app/(auth)/reset-password/page.tsx`, `src/app/page.tsx` (home). Example: `<main className="force-light flex min-h-screen items-center justify-center ...">`.

**Why this works:** `--foreground`, `--input`, `--border`, `--muted-foreground` etc. are CSS custom properties. `.dark` sets them on `<html>`. A class on a *descendant* element (the page's `<main>`) that re-declares the same custom property name takes precedence for everything inside that subtree, because custom property resolution follows normal CSS cascade/specificity — the closest ancestor declaration wins. This is the systemic fix; it does not require touching `Input`, `Label`, `Form`, or any shared component's own source (per the explicit out-of-scope instruction).

**Cleanup required (do not leave both fixes in place):** remove the `className="border-slate-300"` props added directly to the `Input`/`PasswordInput` JSX call sites in `login/page.tsx` and `signup/page.tsx` (4 occurrences total — 2 in login, 3 in signup counting the `fullName`/`email`/`password` fields... actually verify exact count at execution time by grep). These become redundant/competing once `.force-light` re-declares `--input` correctly; leaving both is not harmful but the user explicitly asked for the systemic fix to *replace* the patch, not stack with it.

**Verification gate for this fix:** must be checked with the browser DevTools "Emulate CSS prefers-color-scheme: dark" + the app's own `.dark` class toggled on `<html>` (next-themes), not just visual inspection in light mode — the bug only reproduces when `.dark` is the active ancestor class.

## Typography

No new font-size scale is introduced this phase — sizes (14px/24px) and line-heights reuse exactly what login/signup/home already declare. Font weight is collapsed to exactly 2 values for this phase's scope (see below), which changes 2 roles (form labels, nav link labels) from the codebase's pre-existing `font-medium` (500) to `font-semibold` (600) so the contract does not exceed the 2-weight maximum:

Exactly 2 weights declared for this phase's scope (auth pages + nav): regular (400) for body/input/subtext, semibold (600) for all emphasis (headings, form labels, nav link labels). The codebase elsewhere also uses an intermediate `font-medium` (500) on some pre-existing elements (e.g. current header nav links, "Back to home" link) — those are outside this phase's touched components and are not re-specified here; within the scope of this phase's redesigned/restructured elements, emphasis collapses to one weight (600) so the contract stays at the 2-weight maximum.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / input text / placeholder | 14px (`text-sm`) | 400 (regular) | 1.5 (Tailwind default `leading-normal` on `text-sm`) |
| Form label (`FormLabel`) | 14px (`text-sm`) | 600 (semibold — change from current `font-medium`/inherited weight to `font-semibold` for this phase's redesigned auth pages) | 1.5 |
| Card heading (`h1` "Welcome back" / "Create your account") | 24px (`text-2xl`) | 600 (`font-semibold`) | 1.2 (`tracking-tight`, Tailwind default heading line-height) |
| Card subtext | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Nav link label (sidebar, restructured) | 14px (`text-sm`) | 600 (`font-semibold` — change from current `font-medium` for consistency with the 2-weight contract above) | 1.5 |
