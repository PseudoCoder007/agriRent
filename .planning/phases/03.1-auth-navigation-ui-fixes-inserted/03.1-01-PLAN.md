<objective>
Fix the root cause of dark-mode CSS variable bleed-through on five pages that must always render light regardless of the user's `.dark` theme preference: login, signup, forgot-password, reset-password, and home. Add a single `.force-light` class to `src/app/globals.css` that re-declares every shadcn/Tailwind CSS custom property these pages' components consume, then apply that class to each page's root wrapper. Additionally strip the `dark:bg-slate-900/70 dark:text-slate-50` Tailwind variant classes that a parallel in-progress redesign already added to login/signup's form inputs — `dark:` variants apply independent of any CSS-custom-property override, so they are a second, distinct bleed-through vector that `.force-light` alone cannot neutralize.

Purpose: Per UI-AUTH-01, these pages must stay fully legible (labels, borders, placeholders, dividers — not just the page background) when `.dark` is the active ancestor class on `<html>`. The Phase 2.1 fix only addressed the page background gradient; it never touched the CSS custom properties (`--foreground`, `--input`, `--border`, `--muted-foreground`) that `FormLabel`, `Input`, `RadioGroupItem`, and the "or" divider inherit from `.dark` on `<html>`. Separately, login/signup's form inputs currently carry literal `dark:*` Tailwind classes (added by an in-progress redesign, not by this phase) that override colors specifically when `.dark` is present — these must be removed because they are a structural violation of "always render light," not something `.force-light`'s CSS-variable override can suppress (Tailwind's `dark:` variant compiles to a separate, more-specific selector that wins regardless of custom-property values).

Output: `.force-light` class in globals.css; five page files with the class applied to their root `<main>` wrapper; login/signup inputs with `dark:*` variant classes removed (the rest of their existing redesign styling — `h-11 rounded-xl border-slate-300 bg-slate-50/95 px-4 text-slate-950 shadow-sm placeholder:text-slate-400 focus-visible:border-emerald-500 focus-visible:ring-emerald-500/20` — is kept unchanged, it does not conflict with this phase's goal).
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03.1-01 | Information Disclosure | `.force-light` CSS class | accept | Pure visual/cosmetic CSS custom-property override scoped to a `class` selector — it changes only color token values consumed by `border-color`/`color`/`background-color` CSS properties. It does not change which DOM elements render, what data is fetched, or what is conditionally shown/hidden based on theme. There is no code path where `.force-light` could leak dark-mode-only *content* to a light-only context or vice versa, because no content (only color) is gated by these tokens on these five pages. Confirmed by reading all five page files: none use `dark:` Tailwind variants for content gating (only the now-removed color-only variants on form inputs), and none use theme-conditional rendering logic. |
| T-03.1-02 | Information Disclosure | `dark:*` variant removal on login/signup inputs | accept | Removing `dark:bg-slate-900/70 dark:text-slate-50` only affects rendered input background/text color when `.dark` is active — it does not change validation, submission, or any data path. This is the structural half of the UI-AUTH-01 fix: Tailwind `dark:` variants compile to a separate, independently-specific CSS rule that activates whenever `.dark` is an ancestor, regardless of any custom-property override `.force-light` declares — so removing the variant entirely (rather than trying to out-specificity it) is the only correct fix. |
| T-03.1-SC | Tampering | No new package installs in this plan | accept | This plan modifies only existing first-party files (globals.css, page components). No `npm install` / `pip install` / `cargo add` commands are run. Package Legitimacy Gate not applicable. |
