<objective>
Fix the root cause of dark-mode CSS variable bleed-through on five pages that must always render light regardless of the user's `.dark` theme preference: login, signup, forgot-password, reset-password, and home. Add a single `.force-light` class to `src/app/globals.css` that re-declares every shadcn/Tailwind CSS custom property these pages' components consume, then apply that class to each page's root wrapper. Remove the now-redundant `border-slate-300` per-input patches on login/signup once the systemic fix lands (per 03.1-UI-SPEC.md, the two fixes must not coexist).

Purpose: Per UI-AUTH-01, these pages must stay fully legible (labels, borders, placeholders, dividers — not just the page background) when `.dark` is the active ancestor class on `<html>`. The Phase 2.1 fix only addressed the page background gradient; it never touched the CSS custom properties (`--foreground`, `--input`, `--border`, `--muted-foreground`) that `FormLabel`, `Input`, `RadioGroupItem`, and the "or" divider inherit from `.dark` on `<html>`.

Output: `.force-light` class in globals.css; five page files with the class applied to their root `<main>` wrapper; login/signup pages cleaned of the redundant `border-slate-300` literal patches.
</objective>
