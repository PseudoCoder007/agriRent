# Phase 03.4: Code Review Report

**Reviewed:** 2026-06-28T00:00:00Z
**Depth:** standard
**Files Reviewed:** 13
**Status:** issues_found

## Summary

Reviewed the user profile management feature (view/edit display name, upload/change avatar). The server-side identity derivation pattern is solid: `userId` is consistently re-derived from `supabase.auth.getUser()` in both Server Actions and never accepted from client payloads, matching the project's stated non-negotiable. The `{success, message, data}` response shape is followed correctly everywhere. Zod validation is present on both the profile-fields payload and the avatar file metadata, with the duplication of `imageFileSchema` documented and intentional.

The most significant issue is in the avatars Storage bucket RLS: the `INSERT`/`UPDATE`/`DELETE` policies in `0008_profile_fields.sql` use `(storage.foldername(name))[1] = auth.uid()::text` as the *only* anchor, which is sound, but because the storage path embeds the literal extension from client-supplied data and the **file extension used to build that path is taken from the client-controlled filename rather than derived from the validated MIME type**, a crafted filename can place a non-image extension into Storage at a path that still satisfies the per-user ownership check. This is a content-type/extension-mismatch gap, not an ownership-bypass gap — see CR-01.

There are also a few correctness/maintainability gaps: a `single()` query in the avatar upload's profile read pattern is unaffected (good), but `getAvatarUrl`'s fallback (`Date.now()` when `avatarUpdatedAt` is null) defeats CDN caching for any user whose avatar was set before this column existed or who never re-uploads, and the `phone` PII broad-read tradeoff is documented in the migration but not enforced anywhere in application code — there's no grep-able guard preventing a future page from rendering `profile.phone` for a non-owner.

## Critical Issues

### CR-01: Avatar file extension derived from client-controlled filename, not from validated MIME type

**File:** `src/lib/services/profile.service.ts:107-108`
**Issue:** `uploadAvatar` validates `imageFile.type` (MIME) and `imageFile.size` via `avatarFileSchema`, which is correct defense-in-depth. However, the Storage object path is built from `imageFile.name`, not from the validated `type`:

```ts
const ext = imageFile.name.split(".").pop() ?? "jpg";
const storagePath = `${userId}/avatar.${ext}`;
```

`File.name` is entirely client-controlled and is never validated by `avatarFileSchema` (which only checks `size` and `type`). A request can set `type: "image/jpeg"` (passing Zod and the bucket's `allowed_mime_types` check) while naming the file `avatar.html`, `avatar.svg`, `avatar.php`, or anything else — `ext` then becomes that arbitrary string, and the object lands in the public `avatars` bucket at `${userId}/avatar.html` (or similar) with a public, no-auth-required read URL. Since the bucket is `public: true`, this stored object becomes a publicly reachable URL under the project's own Supabase domain with attacker-chosen extension and (for some browsers/proxies) attacker-influenced content sniffing, even though the actual bytes/content-type are constrained. This is a stored-content/extension-confusion issue: combined with the bucket being public and the path being predictable (`${userId}/avatar.<ext>`), it allows hosting arbitrary-extension files (not arbitrary content-type, since `contentType: imageFile.type` is still sent and Supabase Storage does check `allowed_mime_types` against the upload's declared content-type) at a URL that looks like an image asset. At minimum this breaks the intended invariant "every object in `avatars/` is one of the three image types with a matching, predictable extension," which the rest of the codebase (e.g. `getAvatarUrl`) implicitly assumes.

**Fix:** Derive the extension from the validated MIME type instead of trusting the filename:

```ts
const EXT_BY_MIME: Record<string, string> = {
  "image/jpeg": "jpg",
  "image/png": "png",
  "image/webp": "webp",
};
const ext = EXT_BY_MIME[imageFile.type] ?? "jpg";
const storagePath = `${userId}/avatar.${ext}`;
```

This removes any dependency on client-supplied filename for path construction, closing the gap while keeping the existing fixed-path-per-user overwrite behavior.

## Warnings

### WR-01: `users update own row` RLS policy has no `WITH CHECK`, relies entirely on app-layer column scoping for this phase's writes

**File:** `supabase/migrations/0001_init_schema.sql:74-76` (pre-existing, not modified by this phase) — consumed by `src/lib/services/profile.service.ts:32-40` and `:129-135`
**Issue:** The existing UPDATE policy is `USING ((select auth.uid()) = id)` with no explicit `WITH CHECK`. Postgres falls back to reusing `USING` as the check when `WITH CHECK` is omitted, so this still prevents a user from updating *another user's row* — that part is sound. However, RLS in Postgres is row-level, not column-level: nothing in the database itself prevents an authenticated user from calling `PATCH /rest/v1/users?id=eq.<self>` directly (bypassing `profile.service.ts` entirely) with a body that includes `{"role": "owner"}` or other columns this phase's service layer never intends to expose. `updateProfile()` in `profile.service.ts:34-37` only ever sends `full_name`/`phone` because it spreads from a narrowly-typed `input` object, but that's an application-layer guarantee, not a database-enforced one — a direct REST/PostgREST call against the `users` table with a valid session JWT is a privilege-escalation vector that pre-dates this phase but is newly relevant now that `phone`/`avatar_url`/`avatar_updated_at` are also writable through the same unscoped policy.
**Fix:** Out of this phase's stated scope to fix the underlying policy (it's inherited, not introduced here), but worth flagging for a follow-up: add a trigger or `WITH CHECK` that explicitly forbids changes to `role` (and any other sensitive column) on `UPDATE`, e.g. a `BEFORE UPDATE` trigger that raises if `NEW.role IS DISTINCT FROM OLD.role` unless executed with a service-role bypass. Document this as an accepted-or-tracked risk in `0008_profile_fields.sql`'s comments if not fixed now, similar to how the phone-PII tradeoff is already documented there.

### WR-02: `getAvatarUrl` cache-buster falls back to `Date.now()`, defeating its own purpose for legacy/never-set rows

**File:** `src/lib/services/profile.service.ts:66-74`
**Issue:**
```ts
const version = avatarUpdatedAt ? new Date(avatarUpdatedAt).getTime() : Date.now();
```
The doc comment states the intent is "the browser/CDN never serves a stale cached copy," but `Date.now()` produces a *new* value on every single render call (every page load, every layout render) whenever `avatarUpdatedAt` is null while `avatar_url` is non-null. In current code this branch is actually unreachable in practice today (both columns are always written together in `uploadAvatar`, and the caller only calls `getAvatarUrl` when `avatar_url` is truthy), but the function signature still allows `avatarUpdatedAt: string | null` and is called from 4 separate call sites (`farmer/profile/page.tsx:33`, `owner/profile/page.tsx:33`, `(farmer)/layout.tsx:63`, `(owner)/layout.tsx:61`) all guarded the same way. If a future migration or manual DB edit ever sets `avatar_url` without `avatar_updated_at` (e.g. a backfill script), every render would generate a fresh cache-busting query string, making the avatar effectively uncacheable — the opposite failure mode of "stale," but still a real perf/CDN-defeating bug waiting for a trigger.
**Fix:** Either drop the `Date.now()` fallback (use a fixed epoch sentinel like `0` so the URL is at least stable across renders when the timestamp is missing) or assert the invariant at the type level by making `avatar_url` and `avatar_updated_at` always written together (already true in code) and treating a null `avatar_updated_at` alongside a non-null `avatar_url` as a data-integrity bug to log rather than silently paper over with `Date.now()`.

### WR-03: `phone` PII broad-read tradeoff documented in migration but not enforced or guarded in any reviewed application code

**File:** `supabase/migrations/0008_profile_fields.sql:10-18`; no corresponding guard found in `src/app/(farmer)/farmer/profile/page.tsx`, `src/app/(owner)/owner/profile/page.tsx`, `src/lib/services/profile.service.ts`
**Issue:** The migration comment explicitly accepts that `phone` is readable by any authenticated user via the existing `users select any authenticated` policy, same tradeoff as `email`. That's a reasonable, documented decision for this phase. However, both profile pages select `phone` via `.select("full_name, phone, avatar_url, avatar_updated_at, role, email")` (`farmer/profile/page.tsx:24`, `owner/profile/page.tsx:24`) and pass it straight into `ProfileForm` as `initialPhone={profile.phone}` — which is correct *only* because both queries are scoped with `.eq("id", userData.user.id)`, i.e. each page only ever fetches and renders the caller's own row. There is no equivalent guard or comment anywhere stating "never call this query pattern for another user's id," so a future phase (e.g. a public owner-profile or farmer-contact page) that reuses this same `.select(...)` shape against a different user's id would silently leak phone numbers into a UI, exactly the failure mode the migration comment warns against for "any UI." This phase itself is safe; the warning is about the absence of any guardrail (lint rule, comment-only convention, or a narrower view) that would catch the mistake when it's made later.
**Fix:** Add a one-line comment at the `phone` column's only two read sites (the two profile pages) cross-referencing the migration's warning, e.g. `// phone is broad-read at the DB level (see 0008_profile_fields.sql) — do not reuse this .select() shape against another user's id`. Optionally, consider exposing phone only through a dedicated `profile.service.ts` getter (`getOwnProfile(userId)`) that hard-codes the `.eq("id", userId)` scope server-side instead of duplicating the raw `.select()` string in two page components, which would make a future copy-paste-without-scoping mistake harder to introduce.

### WR-04: `uploadAvatarAction`/`updateProfileAction` revalidate four fixed paths regardless of which role the caller actually has

**File:** `src/app/actions/profile.actions.ts:54-59`, `:101-106`
**Issue:** Both actions unconditionally call all four of `revalidatePath("/farmer")`, `revalidatePath("/owner")`, `revalidatePath("/farmer/profile")`, `revalidatePath("/owner/profile")` regardless of the caller's actual role. A farmer updating their profile will never have cached `/owner` or `/owner/profile` segments to invalidate, so two of the four calls are guaranteed no-ops for any given user. This isn't a bug (Next.js's `revalidatePath` is a no-op for paths that weren't cached) but it is dead, unconditionally-run code that obscures intent — a reader has to know "every user's role-specific paths get revalidated by the other two calls being harmless no-ops" rather than seeing the actual role-scoped invalidation directly.
**Fix:** Either accept the role from the already-fetched `userData`/profile lookup and revalidate only the relevant pair, or leave a one-line comment explaining the four calls are intentionally role-agnostic to avoid an extra DB round-trip just to determine which two paths to skip. Low priority — flagging only because the broad pattern continues into Plan 03 without a comment explaining why all four are always called.

## Info

### IN-01: `ProfileForm`'s `isFieldErrors` type guard does not check value types of `fullName`/`phone`, only key presence

**File:** `src/components/profile/profile-form.tsx:39-45`
**Issue:**
```ts
function isFieldErrors(data: unknown): data is FieldErrors {
  return (
    typeof data === "object" &&
    data !== null &&
    ("fullName" in data || "phone" in data)
  );
}
```
This narrows `data: unknown` to `FieldErrors` based purely on key presence, not on the values actually being `string[] | undefined`. In current usage this is safe because the only producer of this shape is `parsed.error.flatten().fieldErrors` in `profile.actions.ts:45`, which always produces that exact shape. But the guard itself would pass for `{ fullName: 42 }` or `{ fullName: "not an array" }`, and the subsequent `result.data.fullName?.[0]` access (line 67) would then either throw (string indexing returns a char, not undefined, so no throw, but wrong value) or silently misbehave if the action's error shape ever changes. Not a live bug today, but the guard provides weaker safety than its name implies.
**Fix:** Tighten the guard or simply trust the Server Action's return type via TypeScript's `ActionResult` (already typed in `profile.actions.ts:9-13` as `data: unknown`, which is itself part of why this guard exists) — consider typing `ActionResult` generically (`ActionResult<T>`) so `data` doesn't need to be narrowed with a hand-written runtime guard at all.

### IN-02: `email` is read from `profile.email` as a fallback even though `userData.user.email` is the authoritative Auth value

**File:** `src/app/(farmer)/farmer/profile/page.tsx:44`, `src/app/(owner)/owner/profile/page.tsx:44`
**Issue:** `email={userData.user.email ?? profile.email}` — this is defensive and not wrong, but `userData.user.email` being undefined would be unusual for an authenticated Supabase session (email is normally always present unless the project uses phone-only auth, which this project does not appear to). If the two values ever diverge (e.g. user changed their Auth email but `public.users.email` wasn't updated by a corresponding sync), this silently prefers the Auth value without surfacing the staleness anywhere. Informational only — no evidence of an email-sync mechanism being in scope for this phase.
**Fix:** No action required for this phase. If a future "change email" feature is added, ensure `public.users.email` and `auth.users.email` are kept in sync (e.g. via a trigger) so this fallback never actually diverges in practice.

### IN-03: `avatar-upload.tsx` does not reset/clear the avatar preview while `isUploading`, briefly showing the stale image during upload

**File:** `src/components/profile/avatar-upload.tsx:32-54`
**Issue:** While `isUploading` is true, the avatar `<Avatar>` continues to render `currentAvatarUrl` (the previous image) rather than a local object-URL preview of the just-selected file or a loading skeleton. This is a minor UX gap, not a bug — `router.refresh()` on success correctly re-fetches the new `avatar_url` from the server and the image updates. No data integrity or security implication.
**Fix:** Optional UX polish: use `URL.createObjectURL(file)` for an instant local preview during the upload, revoking it on completion. Not required for correctness.
