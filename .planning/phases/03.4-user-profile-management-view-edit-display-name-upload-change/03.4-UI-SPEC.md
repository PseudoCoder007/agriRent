## Entry Point Decision — Profile Access from Nav

**[Default — proceeding without blocking, per orchestrator instruction]**

The Phase 03.1 nav contract (approved, not yet executed in code) restructures both `(farmer)/layout.tsx` and `(owner)/layout.tsx` into 3 vertical zones: top utility row (`ThemeToggle` + `NotificationBell`), middle nav links, bottom-pinned `LogoutButton`. This phase's UI must compose into that structure without contradicting it.

**Decision:** Replace the bottom-zone's bare `LogoutButton` with an account `DropdownMenu` triggered by an avatar+name button. This consolidates identity, profile access, and logout into one conventional "account menu" pattern and reuses the `DropdownMenu` primitive already wired up for `NotificationBell` — no new interaction pattern introduced.

| Zone (per 03.1 contract) | Before this phase | After this phase |
|---|---|---|
| Bottom-pinned (desktop sidebar `mt-auto` zone, mobile Sheet bottom zone) | `LogoutButton` alone | `DropdownMenuTrigger` button: `Avatar` (size `sm`, 24px) + display name (truncated, `text-sm font-semibold`) + chevron-down icon, full-width, `variant="ghost"`, `justify-start` → opens `DropdownMenuContent` with: `DropdownMenuLabel` (full name + email, 2-line), `DropdownMenuSeparator`, `DropdownMenuItem` "Edit profile" (routes to `/farmer/profile` or `/owner/profile`), `DropdownMenuSeparator`, `DropdownMenuItem` "Log out" (destructive variant, wraps existing `LogoutButton`'s server action) |
| Top utility row | `ThemeToggle` + `NotificationBell` | Unchanged — this phase does not touch this zone |
| Middle nav links | Farmer: Dashboard/Browse/Favorites/Chat. Owner: Dashboard/Chat | Unchanged — no new "Profile" link added here (avoids duplicate entry points; the account menu is the single source of profile access) |

**If Phase 03.1's nav restructure has not yet landed in code when this phase executes** (current layout files still show the pre-03.1 horizontal header — confirmed via direct read of `src/app/(farmer)/layout.tsx` and `src/app/(owner)/layout.tsx` at research time): apply the same avatar+name dropdown to the *existing* horizontal header's right-aligned utility cluster, replacing the bare `LogoutButton` in `<div className="hidden items-center gap-2 sm:flex">` (desktop) and the Sheet's bottom `<div className="mt-auto flex flex-col gap-3 border-t ...">` (mobile) — same dropdown contents, different host layout. Executor confirms which nav structure is live at implementation time and adapts the host markup only; the dropdown's internal contract (trigger, items, copy) does not change either way.

**Why not a dedicated "Profile" nav link instead:** would create two entry points to the same destination (the existing role-label text next to the logo is already adjacent to the brand, not a natural second profile-link slot) and adds a 5th/3rd nav-list item that competes with the core task links (Dashboard/Browse/Favorites/Chat) for attention. The account-menu pattern is the more conventional, lower-clutter choice for a "secondary, low-frequency" task like profile editing.

## Profile Page Layout

New route: `/farmer/profile` and `/owner/profile` (or a single shared route resolved per-role server-side — executor's call; both render identical UI). Uses the existing `PageShell` component (`src/components/ui/page-shell.tsx`) for the title/subtitle/actions header pattern already used elsewhere in the app.

| Element | Contract |
|---|---|
| Page header | `PageShell` with `title="Profile"`, no `subtitle` (or `subtitle="Manage your name, photo, and contact details"`), no `actions` |
| Layout container | Single `Card` (`max-w-2xl`, matches `ChatWidget`'s `mx-auto max-w-2xl` precedent), centered under the `PageShell`'s `max-w-5xl` wrapper |
| Card structure | `CardHeader` (no title text needed — avatar block IS the header) → `CardContent` (form fields) → form's own submit row inside `CardContent` (no `CardFooter` needed, consistent with this codebase not using `CardFooter` elsewhere) |
| Avatar block (top of card) | Centered `Avatar` (size `lg`, 40px container but rendered larger via explicit `className="size-24"` override — profile pages conventionally show a bigger avatar than nav/chat contexts) with `AvatarImage` (current photo) / `AvatarFallback` (first letter of display name, `bg-muted text-muted-foreground`, matches existing Avatar primitive defaults). Below the avatar: a `Button variant="outline" size="sm"` labeled "Change photo" that opens a hidden file input (native `<input type="file" accept="image/jpeg,image/png,image/webp" className="hidden">` pattern, NOT a custom Dialog — keeps it a single click, matches the simplicity of the existing equipment-image upload's single-file-input pattern in spirit, though that one is a form field not a profile-photo control) |
| Display name field | `FormField` + `FormLabel` "Display name" + `Input` (controlled, react-hook-form + Zod resolver per CLAUDE.md convention), `FormMessage` for validation errors |
| Phone number field | `FormField` + `FormLabel` "Phone number" + `Input type="tel"` , `FormMessage` for validation errors. Optional field — placeholder "e.g. 9876543210" |
| Email (read-only context) | Show as static text below/near the name field, NOT an editable `Input` — `<p className="text-sm text-muted-foreground">{email}</p>` with a small label "Email" above it (`text-xs font-semibold text-muted-foreground`) — email is Supabase Auth-managed, not part of this phase's edit surface (PROFILE-01/02/03 do not include email editing) |
| Role badge (context only) | Existing `Badge` component, `variant="secondary"`, shows "Farmer" or "Owner" next to/below the name — read-only, not editable (role changes are out of scope) |
| Submit button | `Button` full-width on mobile / auto-width on `sm:`, label "Save changes" (submitting state: "Saving...") — bottom of form, inside `CardContent`, not a floating/sticky footer |
| Photo upload feedback | On successful upload: `Avatar`'s `AvatarImage` updates immediately (optimistic or post-revalidation, executor's call) + `sonner` toast "Profile photo updated". On validation failure (wrong MIME/size): `sonner` toast error, same wording style as `listing.service.ts`'s existing message: "Invalid image: must be JPEG/PNG/WebP under 5MB" |

## Spacing Scale

Declared values (must be multiples of 4) — reused from Phase 03.1's established scale, no new values introduced:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (avatar dropdown chevron, role badge icon if used) |
| sm | 8px | Label-to-input gap (`FormItem`'s existing `gap-2`), dropdown item internal padding |
| md | 16px | Field-to-field spacing in the profile form (`gap-4` on the form, matches auth forms) |
| lg | 24px | Card internal padding (`p-6`, matches `Card` default) |
| xl | 32px | Card padding on `sm:` breakpoint if widened (`sm:p-8`) — optional, only if executor widens beyond default `Card` padding |
| 2xl | 48px | Vertical gap between avatar block and form fields within the card |
| 3xl | 64px | Not used this phase |

Exceptions:
- Avatar dropdown trigger button: full-width touch target in the bottom nav zone, minimum height `h-11` (44px) on mobile to match the existing `LogoutButton`/`ThemeToggle` touch-target convention from Phase 02.1/03.1.
- Profile photo avatar (the large one on the profile page itself, not the nav trigger): `size-24` (96px) — not on the 4px-multiple spacing scale by definition of being a sizing token, not a spacing token; consistent with how `Avatar`'s own `size` prop already departs from the spacing grid (8/10/etc. px sizes) by design.

## Typography

Reuses the project's established 2-weight contract from Phase 03.1 (regular 400 + semibold 600) — no new weight introduced. Sizes reuse the existing `text-sm`/`text-base`/`text-xl` scale already used by `PageShell`/`Card`/auth forms; no new size introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Page title ("Profile", via `PageShell`'s `h1`) | 20px (`text-xl`) | 600 (`font-semibold`) | 1.2 (`tracking-tight`) |
| Form label | 14px (`text-sm`) | 600 (`font-semibold` — matches Phase 03.1's `FormLabel` contract) | 1.5 |
| Body / input text / static email text | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Dropdown trigger name text | 14px (`text-sm`) | 600 (`font-semibold`, matches nav link label weight per 03.1) | 1.5 |
| Dropdown menu label (name + email) | 14px (`text-sm`) name / 12px (`text-xs`) email subtext | 600 name / 400 email | 1.5 |

No 3rd or 4th size is introduced — this phase stays within the 2 sizes already declared (14px, 20px) plus the pre-existing 12px micro-text role (`text-xs`, used for the role badge / email subtext / dropdown email line, same role Phase 03.1 used for "Owner workspace"/"Farmer workspace" subtext) for a total of 3 sizes, within the 3-4 size budget.

|---------|------|
| Primary CTA (save profile) | "Save changes" (submitting: "Saving...") |

| Field label — name | "Display name" |
| Field label — phone | "Phone number" |
| Field label — email (static) | "Email" |
| Phone placeholder | "e.g. 9876543210" |

| Empty state — no profile photo yet | `AvatarFallback` renders the first letter of the display name (or a generic user icon if name is also empty) — existing `Avatar` primitive default behavior, no new empty-state copy needed |
| Success toast — name/phone saved | "Profile updated" |
| Success toast — photo updated | "Profile photo updated" |
| Error toast — invalid image | "Invalid image: must be JPEG, PNG, or WebP under 5MB" (matches existing `listing.service.ts` wording exactly, for consistency) |
| Error toast — save failed (generic) | "Could not update your profile. Please try again." (matches the project's existing service-layer error-message tone, e.g. `listing.service.ts`'s "Could not update equipment listing.") |

No destructive *data-loss* actions exist in this phase's scope (editing name/phone/photo are all reversible/overwritable, logout is non-destructive per 03.1 precedent) — no confirmation dialogs required anywhere in this phase.

## Implementation Notes for Planner (non-UI, flagged for awareness)

- The `public.users` table (per `supabase/migrations/0001_init_schema.sql`) currently has columns `id, email, full_name, role, created_at` — **no `phone` or `avatar_url`/`avatar_path` column exists yet**. A schema migration adding both (plus RLS already covers own-row UPDATE per existing policies — verify at plan time) is a prerequisite for PROFILE-02/PROFILE-03 and is a planning/research concern, not a UI-SPEC concern, but is flagged here so the planner does not discover it late.
- Profile photo storage should follow the same Supabase Storage pattern as `equipment-images` (see `listing.service.ts`): a dedicated bucket (e.g. `profile-photos` or `avatars`), MIME validation via the existing `imageFileSchema` (JPEG/PNG/WebP, 5MB cap) or a near-identical schema, storage path keyed by user id (e.g. `${userId}/${filename}`), public-read bucket policy mirroring `0002_equipment_images_bucket.sql`.
- The nav-visible display name (success criterion 1) requires reading `users.full_name` in both `(farmer)/layout.tsx` and `(owner)/layout.tsx` — both files already fetch the user's `role` from `public.users` in a `.select("role")` query; this becomes `.select("role, full_name, avatar_url")` (or similar) once the schema migration lands, then threaded into the new avatar+dropdown component described above.
