<objective>
Complete PROFILE-02: a user can upload or replace their profile photo from the profile page. The photo is validated server-side (MIME/size, defense-in-depth alongside the bucket's own `allowed_mime_types`), stored at a fixed per-user path with `upsert: true` (so repeat uploads never fail with "Asset Already Exists"), and rendered with a cache-busting query param everywhere the avatar appears (profile page, nav account menu) so the new photo is visible immediately rather than showing a stale CDN-cached copy.

Purpose: This is the second and final vertical slice of Phase 03.4 -- it completes the profile feature by replacing the `{/* Avatar block wired in 03.4-03 */}` placeholder left in Plan 02's `profile-form.tsx` with a real, working upload control, and wires the resulting `avatar_url` into `AccountMenu`'s nav-visible avatar.

</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client file picker -> uploadAvatarAction | Untrusted file bytes/metadata cross this boundary on every upload attempt |
| uploadAvatarAction -> Storage `avatars` bucket | The constructed storage path (`${userId}/avatar.<ext>`) crosses into Postgres RLS evaluation on every write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03.4-07 | Tampering | `uploadAvatar`'s `avatarFileSchema.safeParse` check | mitigate | Re-validated server-side on every call, independent of the `accept="image/jpeg,image/png,image/webp"` HTML attribute on the client file input (a UX hint only, trivially bypassed via a direct POST with a crafted FormData) |
| T-03.4-08 | Tampering / Elevation of Privilege | Crafted storage path attempting to write outside the caller's own `${userId}/` prefix | mitigate | `uploadAvatar` always constructs the path from the session-derived `userId` parameter, never from client input; the `avatars` bucket's `(storage.foldername(name))[1] = auth.uid()::text` RLS policy (Plan 01) is the authoritative, independent backstop even if application code were compromised |
| T-03.4-09 | Denial of Service (minor) | Repeated re-upload to the same fixed path | accept | `upsert: true` is intentional and required -- the fixed-path-per-user design trades "no Storage cleanup needed" for "no upload history retained", an accepted tradeoff per RESEARCH.md Anti-Patterns (no orphaned files, no unbounded storage growth from one user re-uploading repeatedly) |
</threat_model>
