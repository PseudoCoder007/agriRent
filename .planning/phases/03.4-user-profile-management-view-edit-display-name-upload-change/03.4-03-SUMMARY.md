# Phase 03.4 Plan 03: Avatar Upload Summary

**Server-validated avatar upload (fixed per-user Storage path, upsert:true, cache-busted display URL) wired end-to-end through a new AvatarUpload component into the profile page and nav account menu, completing PROFILE-02 and Phase 03.4.**

## Accomplishments
- `profile.service.ts`'s `uploadAvatar(userId, imageFile)` re-validates the file server-side via `avatarFileSchema` (defense in depth alongside the bucket's own MIME/size restriction), uploads to a fixed `${userId}/avatar.<ext>` path with `upsert: true` (required — its absence causes every repeat upload to fail with a 400 "Asset Already Exists"), then writes `avatar_url` + `avatar_updated_at` together in the same call
- `getAvatarUrl(storagePath, avatarUpdatedAt)` builds the public Storage URL and always appends a `?v=` cache-busting query param derived from `avatar_updated_at` (or `Date.now()` as a fallback), mirroring `getEquipmentImageUrl`'s construction pattern but never returning a bare, potentially stale-cached URL
- `uploadAvatarAction(formData)` mirrors `createEquipmentAction`'s FormData/file-guard shape exactly, derives identity only from `supabase.auth.getUser()`, and revalidates the same four paths as `updateProfileAction`
- New `AvatarUpload` client component renders a `size-24` avatar with a hidden native file input and a "Change photo"/"Uploading..." button (no Dialog, per UI-SPEC), calling `uploadAvatarAction` directly and `router.refresh()` on success so the avatar updates without a manual page reload
- `profile-form.tsx`'s `{/* Avatar block wired in 03.4-03 */}` placeholder from Plan 02 is now a real `<AvatarUpload>` render, gaining an `avatarUrl` prop
- Both profile pages and both role layouts now select `avatar_updated_at` alongside `avatar_url` and pass a `getAvatarUrl(...)`-built, cache-busted URL to `ProfileForm`/`AccountMenu` instead of the raw storage path — the raw path alone is not a fetchable URL, so this was required for any avatar to render at all in either location

## Decisions Made
- Stored only the storage path (never a full URL) in `avatar_url`, matching `equipment_images.storage_path`'s existing convention — URLs are always built at read time
- Used `router.refresh()` after a successful upload rather than manually updating client-side avatar state, keeping the Server Component page as the single source of truth for the freshly cache-busted URL
- Reset the file input's value in a `finally` block so selecting the same filename twice in a row still fires `onChange`, satisfying the "second upload never fails" acceptance criterion under manual re-test
