## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Any authenticated client -> public.users SELECT | Existing `0003_users_public_read.sql` policy (`USING (true)`) makes every row's columns readable by any authenticated user — this phase extends that same exposure surface to `phone` |
| Any authenticated client -> storage.objects (avatars bucket) | Write path is gated by `(storage.foldername(name))[1] = auth.uid()::text` — untrusted client input (the requested storage path) crosses this boundary on every upload attempt |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03.4-01 | Information Disclosure | `public.users.phone` column, readable via existing `0003_users_public_read.sql` policy | accept | Documented, deliberate extension of the existing accepted `email`-exposure tradeoff (marketplace owner/farmer coordination); migration includes an explicit SQL comment recording the decision. Mitigated at the application layer in later plans: never render another user's phone number in any UI surface — only the authenticated user's own profile page shows their own phone. |
| T-03.4-02 | Tampering | `storage.objects` INSERT/UPDATE/DELETE on the `avatars` bucket | mitigate | `(storage.foldername(name))[1] = auth.uid()::text` check on every write policy — a crafted path targeting another user's `${userId}/` prefix is rejected at the Postgres RLS layer, independent of whatever path the application code constructs |
| T-03.4-03 | Tampering / stored XSS | Malicious-MIME file disguised as an image (e.g. renamed `.html`/`.svg`) uploaded to the avatars bucket | mitigate | Two independent layers: bucket-level `allowed_mime_types = ARRAY['image/jpeg','image/png','image/webp']` (this migration) + application-level `avatarFileSchema` Zod check (Task 2) re-verified server-side before every upload call in Plan 03 |
| T-03.4-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed this plan — Package Legitimacy Audit in RESEARCH.md confirms zero new dependencies for this entire phase |
