<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client form submission -> updateProfileAction | Untrusted `fullName`/`phone` values cross this boundary on every save |
| Server Action -> profile.service.ts | userId crosses this boundary; must always originate from `supabase.auth.getUser()`, never from the action's input parameter |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03.4-04 | Spoofing | `updateProfileAction` | mitigate | `userId` is derived exclusively from `supabase.auth.getUser()` inside the action -- never accepted as a function parameter from client-controlled input, matching every existing `*.actions.ts` file in this codebase |
| T-03.4-05 | Tampering | `updateProfileSchema` re-validation | mitigate | Server Action re-runs `updateProfileSchema.safeParse(values)` even though `profile-form.tsx` already validates client-side via `zodResolver` -- client validation is UX only and is trivially bypassed via a direct POST |
| T-03.4-06 | Information Disclosure | `AccountMenu`'s `DropdownMenuLabel` rendering the user's own email | accept | Only the authenticated user's own email/name is ever passed into their own AccountMenu instance -- no cross-user data flows through this component |
</threat_model>
