## Summary

This phase requires no new libraries and no new architectural patterns — every primitive needed (react-hook-form + Zod resolver forms, Server Action → service-layer mutation, Supabase Storage bucket-per-resource, `users` RLS) already exists in this codebase with a working precedent. The work is: (1) one additive schema migration adding `phone` and `avatar_url` columns to `public.users`, regenerating `types/database.ts`; (2) a new `avatars` Storage bucket + policies, modeled directly on `0002_equipment_images_bucket.sql`; (3) a `profile.service.ts` + `profile.actions.ts` pair following the exact shape of `listing.service.ts`/`listing.actions.ts`; (4) a profile page built with the same `Form`/`FormField` + `react-hook-form` + `zodResolver` pattern already used on the signup page; (5) reading `full_name`/`avatar_url` into both role layouts where `role` is already being selected.

The one real domain risk is **CDN/browser caching of the avatar's public URL after a same-path upload** — Supabase's storage CDN can serve a stale cached image for up to ~60 seconds after a same-path overwrite, a known and documented behavior, not a bug in this codebase. The plan must account for cache-busting (append `?t=<updated_at>` to the rendered URL) or this phase will visibly fail its own success criterion 3 ("renders wherever their profile/avatar is shown") immediately after upload, even though the backend write succeeded.

The existing `users update own row` RLS UPDATE policy (from `0001_init_schema.sql`) already covers this phase's write requirement — **no new RLS policy is needed on `public.users` itself.** Only the new Storage bucket needs its own RLS policies (INSERT/UPDATE/DELETE scoped to the user's own `auth.uid()` path prefix), exactly mirroring the equipment-images bucket policies.

**Primary recommendation:** Add `phone text` and `avatar_url text` nullable columns to `public.users` via a new additive migration; create a public-read `avatars` Storage bucket scoped by `${userId}/` path prefix; build `profile.service.ts`/`profile.actions.ts` mirroring `listing.service.ts`/`listing.actions.ts` exactly; reuse the signup page's `Form` + `react-hook-form` + `zodResolver` pattern for the profile page; cache-bust the avatar URL on render using a stored `updated_at` (or similar) timestamp query param.

### Pattern 1: Service-layer mutation with re-verified identity (existing codebase convention)
**What:** Every mutating service function takes the authenticated `userId` as an explicit parameter (never reads it from the input payload), re-checks ownership/identity against the DB before writing, and returns `{ success, message, data }`.
**When to use:** Every write in this phase (name/phone update, avatar upload).
**Example:**
```typescript
// Source: src/lib/services/listing.service.ts (existing pattern, applied to profile.service.ts)
export async function updateProfile(
  userId: string,
  input: UpdateProfileInput
): Promise<ServiceResult<Tables<"users">>> {
  const supabase = await createClient();

  const { data: updated, error } = await supabase
    .from("users")
    .update({
      ...(input.fullName !== undefined && { full_name: input.fullName }),
      ...(input.phone !== undefined && { phone: input.phone }),
    })
    .eq("id", userId)
    .select()
    .single();

  if (error || !updated) {
    console.error("profile.service.updateProfile: update failed", error);
    return {
      success: false,
      message: "Could not update your profile. Please try again.",
      data: null,
    };
  }

  return { success: true, message: "Profile updated", data: updated };
}
```

### Pattern 2: Storage upload with bucket-prefix-scoped RLS (existing codebase convention)
**What:** Storage object path is always `${userId}/...` so the bucket's own RLS policy (`(storage.foldername(name))[1] = auth.uid()::text`) is the authoritative write-scope guard, independent of the application-layer `ownerId`/`userId` parameter check.
**When to use:** Avatar upload.
**Example:**
```typescript
// Source: src/lib/services/listing.service.ts createEquipment() (adapted)
const AVATAR_BUCKET = "avatars";

export async function uploadAvatar(
  userId: string,
  imageFile: File
): Promise<ServiceResult<string>> {
  const imageCheck = imageFileSchema.safeParse({
    size: imageFile.size,
    type: imageFile.type,
  });
  if (!imageCheck.success) {
    return {
      success: false,
      message: "Invalid image: must be JPEG, PNG, or WebP under 5MB",
      data: null,
    };
  }

  const supabase = await createClient();
  const ext = imageFile.name.split(".").pop() ?? "jpg";
  // Fixed filename per user (not per-upload) — overwrite-in-place via upsert,
  // so old avatar object never orphans in storage and avatar_url never needs
  // to change shape, only its cache-busting query param.
  const storagePath = `${userId}/avatar.${ext}`;

  const { error: uploadError } = await supabase.storage
    .from(AVATAR_BUCKET)
    .upload(storagePath, imageFile, {
      contentType: imageFile.type,
      upsert: true, // REQUIRED — default upload() rejects with 400 "Asset Already Exists" on a repeat path
    });

  if (uploadError) {
    console.error("profile.service.uploadAvatar: upload failed", uploadError);
    return {
      success: false,
      message: "Could not upload your photo. Please try again.",
      data: null,
    };
  }

  const { error: updateError } = await supabase
    .from("users")
    .update({ avatar_url: storagePath })
    .eq("id", userId);

  if (updateError) {
    console.error("profile.service.uploadAvatar: users update failed", updateError);
    return {
      success: false,
      message: "Photo uploaded, but could not save to your profile.",
      data: null,
    };
  }

  return { success: true, message: "Profile photo updated", data: storagePath };
}
```
**Note on `avatar_url` column naming:** Store the **storage path** (`${userId}/avatar.jpg`), not a pre-built public URL, in the `avatar_url` column — exactly how `equipment_images.storage_path` works (see `getEquipmentImageUrl()` in `listing.service.ts`, which builds the URL on read, not on write). This avoids baking the Supabase project URL into stored data (breaks if the project URL ever changes) and keeps the cache-busting query param appendable at render time without re-writing the DB row.

### Anti-Patterns to Avoid
- **Trusting the client-submitted `avatar_url` or `userId` in any Server Action:** Always derive `userId` from `supabase.auth.getUser()` inside the Server Action, exactly like every existing `*.actions.ts` file does. Never accept it as a function parameter from client-controlled input.
- **Building the public avatar URL at write-time and storing it in the DB column:** Store the storage path only (see Pattern 2 note); build the full public URL at read-time, exactly mirroring `getEquipmentImageUrl()`.
- **Per-upload unique filenames for avatars (e.g. `avatar-${timestamp}.jpg`):** This avoids the cache problem but orphans every previous avatar file in Storage forever (no cleanup mechanism exists in this codebase). Prefer a fixed path (`${userId}/avatar.<ext>`) with `upsert: true` + cache-busting on read, not on write.
- **Forgetting `upsert: true` on the avatar upload call:** Without it, every "Change photo" after the first throws a 400 "Asset Already Exists" error from Supabase Storage — this is the single most likely first bug encountered when building this feature.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Phone number format validation | A hand-rolled multi-branch regex covering every country | A single Zod `.regex()` scoped to the project's realistic user base (10-digit Indian mobile numbers, optionally `+91` prefixed) | This is an MVP for an Indian agri-rental context (per CLAUDE.md/PROJECT.md framing); full i18n phone parsing (`libphonenumber-js`) is scope creep for v1 — only add it if a real requirement for international users surfaces |
| Image MIME/size validation | A custom file-signature sniffer | Reuse the existing `imageFileSchema` from `equipment.schema.ts` (or an identical copy in `profile.schema.ts`) + the Storage bucket's own `allowed_mime_types`/`file_size_limit` | Defense-in-depth pattern already established in this codebase (T-03-03); duplicating proven validation logic via copy is acceptable per the "max 500 lines/file" convention, a shared module is also fine if the planner prefers DRY |
| CDN cache invalidation for replaced avatar | A custom cache-purge API call to Supabase | Append a cache-busting query param (`?v=<updated_at timestamp>`) to the rendered URL, sourced from the `users` row's own timestamp (or a new `avatar_updated_at` column if `updated_at` doesn't exist on `users` yet — it does not, per the current schema) | Supabase's CDN propagation is async (up to ~60s) and there is no synchronous purge API exposed to client code; query-param busting is the documented, supported workaround [CITED: github.com/orgs/supabase/discussions/5737] |

**Key insight:** This phase has zero genuinely novel technical problems — it is a structural composition of three patterns (schema migration, service-layer CRUD, Storage-bucket-per-resource) that already exist three times each in this codebase (equipment, bookings, equipment-images). The only non-obvious domain risk is the CDN caching behavior, which is an infrastructure quirk, not a code design problem.

## Common Pitfalls

### Pitfall 1: Avatar appears unchanged after upload due to CDN/browser caching
**What goes wrong:** User uploads a new photo, the Storage write succeeds, the DB `avatar_url` column updates, but the rendered `<img>`/`<Image>` still shows the old photo — because both the browser cache and Supabase's CDN cache the object by its URL path, and a same-path overwrite doesn't always immediately invalidate.
**Why it happens:** Supabase Storage's public URLs are served through a CDN layer; cache invalidation on overwrite is asynchronous (documented up to ~60 seconds), and browsers separately cache image responses by URL. [CITED: github.com/orgs/supabase/discussions/5737, github.com/supabase/storage/issues/120]
**How to avoid:** Append a cache-busting query parameter derived from a timestamp (e.g. `?v=<users.created_at-or-new-column>`) to every rendered avatar URL — never render the bare storage public URL directly. The UI-SPEC's "Photo upload feedback" row says "Avatar's AvatarImage updates immediately (optimistic or post-revalidation, executor's call)" — the cache-busting param is required regardless of which option the executor picks, or the visible bug will reproduce even with `router.refresh()`/`revalidatePath()`.
**Warning signs:** Manual QA where "Change photo" appears to silently do nothing on a second/third attempt, or works in an incognito window but not the original browser tab.

### Pitfall 2: Repeat avatar upload fails with "Asset Already Exists" (400)
**What goes wrong:** First avatar upload succeeds; uploading a replacement photo to the same path throws a Supabase Storage 400 error.
**Why it happens:** `supabase.storage.from(bucket).upload()` defaults to rejecting writes to an existing path; `upsert: true` (or the `x-upsert` header) must be explicitly passed to allow overwrite. [CITED: supabase.com/docs/reference/javascript/storage-from-update]
**How to avoid:** Always pass `{ upsert: true }` in the `.upload()` call inside `profile.service.ts`'s avatar function — this is the single most important implementation detail to get right and the most likely thing to be missed if copy-pasting `createEquipment()`'s upload call (which does NOT use upsert, because equipment images are never overwritten at the same path).
**Warning signs:** Works once per test user in manual QA, then fails on every subsequent attempt by the same user — a strong signal this exact bug is present.

### Pitfall 3: `types/database.ts` drifts from the new schema, causing silent `any`-typed reads
**What goes wrong:** The migration adds `phone`/`avatar_url` columns, but `types/database.ts` is not regenerated — TypeScript still thinks `Tables<"users">` lacks those fields, and either compile errors appear (best case) or the code falls back to loose typing (worse case, silent bugs).
**Why it happens:** `supabase gen types typescript --linked` (per CLAUDE.md's explicit instruction "Run ... after every schema change") is a manual step, easy to forget when the schema change feels small (two nullable columns).
**How to avoid:** Plan must include an explicit task/step to regenerate `types/database.ts` immediately after the migration is applied, before writing any service-layer code that reads/writes the new columns. This phase's plan-checker should verify this task exists given CLAUDE.md's standing instruction.
**Warning signs:** TypeScript errors referencing `Property 'phone' does not exist on type ...` when writing `profile.service.ts`.

### Pitfall 4: Forgetting that Phase 03.1's nav restructure has NOT landed in code yet
**What goes wrong:** A plan or executor assumes `(farmer)/layout.tsx`/`(owner)/layout.tsx` already have the 3-zone vertical structure described in the Phase 03.1 UI-SPEC (top utility row / middle nav / bottom-pinned logout), and writes the new account-menu component against that assumed structure.
**Why it happens:** The Phase 03.1 UI-SPEC document exists and is approved, but its code has not been executed — confirmed by direct read of both layout files during this research pass (2026-06-28): both still use the original horizontal header (`<header className="flex items-center justify-between ...">` with a `hidden sm:flex` desktop cluster and a `Sheet` for mobile), not the vertical 3-zone sidebar described in 03.1's contract. Both files also show as currently modified/uncommitted in `git status`, meaning they may be mid-edit by a concurrent or very recent change.
**How to avoid:** This phase's own UI-SPEC already anticipates this exact scenario and gives an explicit fallback instruction ("If Phase 03.1's nav restructure has not yet landed in code when this phase executes: apply the same avatar+name dropdown to the existing horizontal header's right-aligned utility cluster..."). The plan MUST have the executor re-check the live state of both layout files at execution time (not trust this research snapshot, since the files are actively being modified) and branch accordingly — do not hardcode an assumption about which layout shape exists.
**Warning signs:** New `AccountMenu` component renders in the wrong DOM position or breaks layout because it was built assuming a sidebar container that doesn't exist yet.

### Pitfall 5: RLS UPDATE policy has no `WITH CHECK` clause — verify it still behaves correctly with new columns
**What goes wrong:** `0001_init_schema.sql`'s `"users update own row"` policy only has a `USING` clause (`(select auth.uid()) = id`), no `WITH CHECK`. For a same-row, non-id-changing UPDATE (which is all this phase does), this is sufficient — but it's worth explicitly verifying in the plan/tests that a user truly cannot UPDATE a row where `id != auth.uid()` even via a crafted payload, since the policy was written before `phone`/`avatar_url` existed.
**Why it happens:** RLS policies are column-agnostic by default (row-level, not column-level) — adding new columns to a table never requires a new policy unless you want to restrict which *columns* a role can write, which Postgres RLS does not natively support without `GRANT`/column-privilege tricks (overkill here).
**How to avoid:** No new policy needed (confirmed — see Sources). Add a defense-in-depth test or manual verification step: attempt `UPDATE users SET phone = 'x' WHERE id = '<other-user-id>'` as an authenticated non-admin user and confirm 0 rows affected, mirroring this codebase's existing pattern of double-checking ownership in the service layer (`profile.service.ts`'s `updateProfile` should also explicitly use `.eq("id", userId)` from the session, not trust RLS alone, consistent with `listing.service.ts`'s `updateEquipment` re-verifying `existing.owner_id !== ownerId` before writing).

## Code Examples

### Migration adding phone/avatar_url to users + avatars bucket
```sql
-- Source: pattern from supabase/migrations/0002_equipment_images_bucket.sql, adapted
-- supabase/migrations/0008_profile_fields.sql

-- 1. Additive columns — both nullable, no backfill needed (existing rows get NULL).
ALTER TABLE public.users ADD COLUMN phone text;
ALTER TABLE public.users ADD COLUMN avatar_url text;

-- 2. avatars bucket — public-read (so the storage path can be turned into a
--    public URL the same way getEquipmentImageUrl() does), write-scoped to
--    each user's own ${userId}/ path prefix.
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'avatars',
  'avatars',
  true,
  5242880, -- 5MB, matches imageFileSchema
  ARRAY['image/jpeg', 'image/png', 'image/webp']
)
ON CONFLICT (id) DO UPDATE SET
  public = EXCLUDED.public,
  file_size_limit = EXCLUDED.file_size_limit,
  allowed_mime_types = EXCLUDED.allowed_mime_types;

CREATE POLICY "Public read access to avatars"
ON storage.objects FOR SELECT
TO authenticated
USING (bucket_id = 'avatars');

CREATE POLICY "Users can upload their own avatar"
ON storage.objects FOR INSERT
TO authenticated
WITH CHECK (
  bucket_id = 'avatars'
  AND (storage.foldername(name))[1] = auth.uid()::text
);

CREATE POLICY "Users can update their own avatar"
ON storage.objects FOR UPDATE
TO authenticated
USING (
  bucket_id = 'avatars'
  AND (storage.foldername(name))[1] = auth.uid()::text
);

CREATE POLICY "Users can delete their own avatar"
ON storage.objects FOR DELETE
TO authenticated
USING (
  bucket_id = 'avatars'
  AND (storage.foldername(name))[1] = auth.uid()::text
);

-- NOTE: no new RLS policy needed on public.users itself — the existing
-- "users update own row" policy from 0001_init_schema.sql already covers
-- UPDATE of any column on the caller's own row, including these two new ones.
```

## Open Questions

1. **What is the cache-busting timestamp source for the avatar URL?**
   - What we know: `public.users` currently has no `updated_at` column (only `created_at`); the avatar upload write happens via `profile.service.ts`'s `uploadAvatar()`.
   - What's unclear: Whether to (a) add an `updated_at timestamptz` column + trigger to `public.users` in the same migration, (b) add a narrower `avatar_updated_at` column written only by `uploadAvatar()`, or (c) avoid a DB-sourced timestamp entirely and cache-bust client-side using `Date.now()` captured at the moment of a successful upload response (works for the uploading user's own immediate view via `router.refresh()`, but does NOT help other users/pages that load the avatar later from a server-rendered context, e.g. another farmer viewing this owner's profile in a future feature — out of scope for this phase, but the chosen mechanism should not foreclose it cheaply).
   - Recommendation: Add `avatar_url text` AND store the upload's storage path with a version-suffix-free cache-busting strategy via a narrow `avatar_updated_at timestamptz` column, set to `now()` inside `uploadAvatar()`'s same UPDATE statement (no trigger needed since it's the same write). This is the smallest correct fix and matches the existing migration style of additive, single-purpose columns. The planner should make this call explicit in the plan rather than leaving it to the executor.

2. **Single shared profile route vs. two role-scoped routes (`/farmer/profile` + `/owner/profile`)?**
   - What we know: UI-SPEC explicitly defers this to "the executor's call" — both render identical UI.
   - What's unclear: Whether the codebase's existing route-group convention (`(farmer)`/`(owner)` route groups, confirmed via `layout.tsx` reads) makes one approach clearly more consistent with existing patterns.
   - Recommendation: Given both `(farmer)` and `(owner)` are separate route groups with separate layouts (confirmed: `farmer/dashboard`, `owner/dashboard` are distinct nested paths, not a shared dynamic route), the path-of-least-resistance is two thin page files (`(farmer)/farmer/profile/page.tsx`, `(owner)/owner/profile/page.tsx`) that both render a single shared `<ProfileForm>` client component — avoids inventing a new shared-route pattern this codebase doesn't otherwise use, while still avoiding UI duplication. The planner should lock this in rather than leave it open at execution time, since it affects routing decisions in the new `AccountMenu` dropdown's "Edit profile" link target.

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|----------------------|
| Forged `userId` in a direct POST to a Server Action (bypassing the React form) | Spoofing | `userId` is never accepted as a Server Action parameter from client input — always derived server-side from `supabase.auth.getUser()`, exactly like every existing action in this codebase |
| Cross-user avatar overwrite via a crafted Storage path (e.g. uploading to `<other-user-id>/avatar.jpg`) | Tampering / Elevation of Privilege | Storage bucket RLS policy's `(storage.foldername(name))[1] = auth.uid()::text` check rejects any upload whose path prefix doesn't match the caller's own `auth.uid()` — this is the authoritative guard, independent of whatever path the application code constructs |
| Oversized or malicious-MIME file upload (e.g. an `.html`/`.svg` file disguised with a renamed extension to attempt stored-XSS via a rendered "image") | Tampering / potential stored XSS | Two independent layers: (1) `avatarFileSchema`/`imageFileSchema` Zod check on `file.type` server-side before upload is attempted; (2) the Storage bucket's own `allowed_mime_types` array (`ARRAY['image/jpeg','image/png','image/webp']`) rejects the upload at the Supabase Storage layer even if the application-layer check were somehow bypassed — same defense-in-depth pattern as `equipment-images` |
| RLS bypass attempt by reading another user's `phone`/`avatar_url` via a crafted query | Information Disclosure | The existing `"users select any authenticated (public profile fields)"` policy from `0003_users_public_read.sql` already makes the entire `users` row readable by any authenticated user (documented intentional tradeoff for the marketplace's owner-name-display requirement) — this means `phone` will also be readable by any authenticated user once added, not just the row's own owner. **This is a pre-existing accepted tradeoff being extended to a new, more sensitive column (phone number) and should be flagged explicitly to the user/planner as a discretionary decision**, not silently inherited. See Open Question equivalent below. |

**IMPORTANT FLAG for planner attention:** Because the existing `0003_users_public_read.sql` policy is row-level (`USING (true)` for any authenticated SELECT), adding a `phone` column means **any logged-in farmer or owner can read any other user's phone number** by querying `public.users`, not just their own. This is consistent with how `email` is already exposed today (the migration's own comment acknowledges "Email is also exposed by this broader policy... but is never rendered in any equipment-facing UI"), but a phone number is a more actively-misusable PII field (enables unsolicited calls/SMS) than an email sitting unused in a row. The plan should either (a) explicitly accept this tradeoff and ensure the application layer simply never queries/renders other users' phone numbers in any UI (the application-layer mitigation already used for email), or (b) consider a Postgres view / column-level security workaround if stricter phone privacy is desired. This is a discretionary product decision, not a research-resolvable technical question — flagging for the planner/user rather than assuming.
