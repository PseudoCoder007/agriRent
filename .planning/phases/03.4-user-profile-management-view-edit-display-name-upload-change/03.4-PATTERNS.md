### `supabase/migrations/0008_profile_fields.sql` (migration, batch)

**Analog:** `supabase/migrations/0002_equipment_images_bucket.sql` (bucket creation + storage RLS) and `supabase/migrations/0001_init_schema.sql` (existing `users update own row` policy — no new policy needed there)

**Bucket + storage RLS pattern** (`0002_equipment_images_bucket.sql` lines 13-61):
```sql
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'equipment-images',
  'equipment-images',
  true,
  5242880, -- 5MB
  ARRAY['image/jpeg', 'image/png', 'image/webp']
)
ON CONFLICT (id) DO UPDATE SET
  public = EXCLUDED.public,
  file_size_limit = EXCLUDED.file_size_limit,
  allowed_mime_types = EXCLUDED.allowed_mime_types;

CREATE POLICY "Public read access to equipment images"
ON storage.objects FOR SELECT
TO authenticated
USING (bucket_id = 'equipment-images');

CREATE POLICY "Owners can upload their own equipment images"
ON storage.objects FOR INSERT
TO authenticated
WITH CHECK (
  bucket_id = 'equipment-images'
  AND public.is_owner()
  AND (storage.foldername(name))[1] = auth.uid()::text
);

CREATE POLICY "Owners can update their own equipment images"
ON storage.objects FOR UPDATE
TO authenticated
USING (
  bucket_id = 'equipment-images'
  AND (storage.foldername(name))[1] = auth.uid()::text
);
```
**Note:** the equipment-images INSERT policy adds `AND public.is_owner()` (a role gate) — the new `avatars` bucket policy must NOT add an equivalent role gate (any authenticated user, farmer or owner, can upload their own avatar), so drop that clause and keep only the `(storage.foldername(name))[1] = auth.uid()::text` ownership check, matching RESEARCH.md's Code Examples section exactly.

**Existing UPDATE policy already covering new `users` columns** (`0001_init_schema.sql` — confirmed by RESEARCH.md Pitfall 5): the existing `"users update own row"` policy (`USING (select auth.uid()) = id`) requires no companion migration edit — RLS is row-level, not column-level. Only `ALTER TABLE` + bucket/storage-policy statements are new SQL in `0008`.

**Public-read exposure precedent to follow, not silently extend without flagging** (`0003_users_public_read.sql` lines 1-17): the existing `users select any authenticated` policy (`USING (true)`) already exposes `email` to every authenticated user; adding `phone` extends this same exposure. RESEARCH.md flags this explicitly as a discretionary product decision — the migration itself needs no change, but the plan must record the decision (accept tradeoff + never render other users' phone in UI, mirroring how `email` is handled today).

**CRUD update pattern with re-verified ownership** (`updateEquipment`, lines 155-209) — apply directly to `updateProfile(userId, input)`:
```typescript
export async function updateEquipment(
  equipmentId: string,
  ownerId: string,
  input: UpdateEquipmentInput
): Promise<ServiceResult<EquipmentRow>> {
  const supabase = await createClient();
  const { data: existing } = await supabase
    .from("equipments")
    .select("owner_id")
    .eq("id", equipmentId)
    .single();

  if (!existing || existing.owner_id !== ownerId) {
    return { success: false, message: "You do not have permission to edit this listing", data: null };
  }

  const { data: updated, error: updateError } = await supabase
    .from("equipments")
    .update({
      ...(input.title !== undefined && { title: input.title }),
      // partial-update spread pattern — apply same for full_name/phone
    })
    .eq("id", equipmentId)
    .select()
    .single();

  if (updateError || !updated) {
    console.error("listing.service.updateEquipment: update failed", updateError);
    return { success: false, message: "Could not update equipment listing.", data: null };
  }

  return { success: true, message: "Equipment updated", data: updated };
}
```
For `updateProfile`, the "ownership re-verify" step collapses to `.eq("id", userId)` directly on the UPDATE (no separate existence check needed since a user always has exactly one row matching their own `auth.uid()`), per RESEARCH.md Pattern 1's exact code.

**File-upload pattern with bucket constant + URL builder** (lines 22-33, 107-124):
```typescript
const EQUIPMENT_BUCKET = "equipment-images";

export function getEquipmentImageUrl(storagePath: string): string {
  const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL!;
  return `${supabaseUrl}/storage/v1/object/public/${EQUIPMENT_BUCKET}/${storagePath}`;
}

const storagePath = `${ownerId}/${equipment.id}/${imageFile.name}`;

const { error: uploadError } = await supabase.storage
  .from(EQUIPMENT_BUCKET)
  .upload(storagePath, imageFile, { contentType: imageFile.type });
```
**Critical deviation required for avatars** (per RESEARCH.md Pitfall 2 and Pattern 2): equipment images use a fresh per-upload path (no `upsert`), but avatars MUST use a fixed `${userId}/avatar.<ext>` path with `{ upsert: true }` explicitly passed, or every repeat "Change photo" throws a 400. Store the storage path (not a built URL) in `avatar_url`, building the public URL at read-time via a `getAvatarUrl(storagePath)` function mirroring `getEquipmentImageUrl()` exactly, plus a cache-busting query param sourced from a new `avatar_updated_at` column (RESEARCH.md Open Question 1's recommendation).

**Error-handling/console.error convention** (consistent across all functions, e.g. lines 96-104, 192-202, 241-251): every failure path logs `console.error("<file>.<function>: <stage> failed", error)` then returns a user-facing generic message — never leaks raw Postgres/Storage error text to the client.

```typescript
export async function updateEquipmentAction(
  equipmentId: string,
  formData: FormData
): Promise<ActionResult> {
  const supabase = await createClient();
  const { data: userData } = await supabase.auth.getUser();

  if (!userData.user) {
    return { success: false, message: "You must be logged in to edit a listing.", data: null };
  }

  const parsed = updateEquipmentSchema.safeParse({
    title: formData.get("title") || undefined,
    // ...
  });

  if (!parsed.success) {
    return {
      success: false,
      message: "Please correct the highlighted fields",
      data: parsed.error.flatten().fieldErrors,
    };
  }

  const result = await listingService.updateEquipment(
    equipmentId,
    userData.user.id,
    parsed.data
  );

  if (result.success) {
    revalidatePath("/browse");
    revalidatePath(`/equipment/${equipmentId}`);
  }

  return result;
}
```
For `updateProfileAction(input)`: accept typed JSON values (not FormData) since there is no file in this action — matches RESEARCH.md Pattern 3's "submits typed values (not raw FormData) to a Server Action for non-file fields." Revalidate both profile page paths AND the layout paths (`/farmer`, `/owner` root layouts re-render nav) so the new `full_name`/`avatar_url` show up immediately in the account-menu dropdown.
