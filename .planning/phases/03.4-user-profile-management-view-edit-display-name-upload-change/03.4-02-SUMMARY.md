# Phase 03.4 Plan 02: Profile Service Layer + Account Menu + Profile Pages Summary

**Server-validated updateProfile mutation wired end-to-end through a new account-menu dropdown and two role-scoped profile pages, replacing the bare LogoutButton in both nav layouts with the user's real display name.**

## Accomplishments
- `profile.service.ts`'s `updateProfile(userId, input)` writes `full_name`/`phone` to `public.users`, always scoped to `.eq("id", userId)` with `userId` supplied only by the caller's own session-derived value — never trusted from client input
- `profile.actions.ts`'s `updateProfileAction(values)` derives identity exclusively from `supabase.auth.getUser()`, re-validates with `updateProfileSchema.safeParse` even though the client already validates via `zodResolver`, and revalidates both role roots plus both profile pages on success
- `ProfileForm` renders display name + phone fields (react-hook-form + Zod) plus a static read-only email block, submits via `updateProfileAction`, and splits error feedback: inline `FormMessage` for server-returned field errors, `sonner` toast for generic failures and the "Profile updated" success case
- `AccountMenu` replaces the bare `LogoutButton` in both `(farmer)/layout.tsx` and `(owner)/layout.tsx` (desktop utility cluster and mobile Sheet bottom zone) with an avatar+name dropdown trigger that opens to an "Edit profile" link and a destructive "Log out" item, reusing the existing `logOutAction`/`useTransition` wiring inline
- New `/farmer/profile` and `/owner/profile` routes render `ProfileForm` pre-filled with the authenticated user's live `full_name`/`phone`/`email` from `public.users`
- Both role layouts now select `role, full_name, avatar_url` (previously `role` only) so the nav-visible name and avatar are real data, not a bare role label

## Decisions Made
- Phone empty string normalized to `null` at the service layer rather than stored as `""`, keeping "no phone" semantically distinct from "explicitly cleared to empty string"
- `AccountMenu` computes `displayName = fullName?.trim() || email.split("@")[0] || "Account"` so the nav never renders blank/undefined text even before a user sets their name
- Used base-ui's `render` prop (already an established pattern via `SheetTrigger render={<Button .../>}` in both layouts) to swap `DropdownMenuTrigger`'s rendered element to `Button` and `DropdownMenuItem`'s to `Link`, rather than wrapping children imperatively
- Committed the pre-existing uncommitted Phase 03.1 nav-restructure changes already present in `(farmer)/layout.tsx`/`(owner)/layout.tsx` together with this plan's `AccountMenu` wiring, since both touch the same lines in the same files and could not be cleanly split into separate commits
