<objective>
Enable bookings to reach their terminal states (completed or cancelled) per BOOK-05. The schema-level trigger was already added in Plan 02-01; this plan builds the service-layer VALID_TRANSITIONS guard, the farmer-scoped RLS policy so farmers can UPDATE their own bookings, two new service functions (completeBooking, cancelBooking), their Server Actions, and the dashboard UI buttons for both roles.

This plan was split from the original 02-03 per gsd-plan-checker scope_sanity feedback — it touches only booking lifecycle files, not favorites.

Purpose: Bookings no longer stop at approved/rejected; they can reach a true terminal state with server-enforced transitions on every code path.
Output: Farmer can self-cancel pending/approved bookings; owner can mark completed or cancel approved bookings; notification created on each transition.
</objective>

    -- Allow the farmer who made the booking to update their own row, needed for
    -- self-cancellation of pending/approved bookings. Coexists with the existing
    -- owner-only UPDATE policy from 0001_init_schema.sql — Postgres ORs multiple
    -- UPDATE policies together, so adding this does not weaken the owner policy.

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> completeBookingAction/cancelBookingAction | A forged `bookingId` belonging to another owner or farmer is the primary untrusted input; ownership re-verification inside each service function rejects mismatches before any write |
| Browser -> cancelBookingAction (farmer calling) | Farmer must only be able to cancel their OWN booking (farmer_id match), not another farmer's booking they have no relationship with |
