<objective>
Give owners full control over their own listings (edit, soft-delete) and give farmers server-side category/location filtering on the browse page — both per EQUIP-02, EQUIP-03, EQUIP-05. This is one vertical slice end-to-end: UI form -> Server Action -> service layer -> RLS-respecting Supabase write/read, for both the edit/delete flow and the filter flow.

Purpose: Owners currently have create-only listings (Phase 1); farmers currently see an unfiltered flat list. This plan completes the equipment-management half of Phase 2's goal.
Output: A working `/equipment/[id]/edit` page (owner-only, ownership re-verified server-side), a soft-delete path that preserves booking history, and a filter bar on `/browse` that narrows results via real database queries.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> updateEquipmentAction/deleteEquipmentAction | A forged `equipmentId` belonging to another owner is the primary untrusted input crossing this boundary |
| Browser -> /browse searchParams | Category/location filter values are fully attacker-controlled URL params; must be passed through parameterized query-builder calls, never string-concatenated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-02-04 | Elevation of Privilege | `updateEquipmentAction` / `listingService.updateEquipment` | mitigate | Service layer re-fetches `equipment.owner_id` and compares to the session's user id BEFORE any `.update()` call; RLS `"equipments update own as owner"` policy (already live from Phase 1) is the second independent gate even if the service-layer check is bypassed via a direct Supabase client call |
| T-02-05 | Elevation of Privilege | `deleteEquipmentAction` / `listingService.softDeleteEquipment` | mitigate | Same ownership re-verification pattern as T-02-04; RLS update policy (soft-delete is an UPDATE, not a DELETE) is the second gate |
| T-02-06 | Information Disclosure | Soft-deleted equipment reachable by direct URL | mitigate | `getEquipmentById` applies `.is('deleted_at', null)`, causing `notFound()` for any caller (owner or not) visiting `/equipment/[id]` for a soft-deleted row; only the owner's dashboard (`getEquipmentByOwner`, unfiltered) can still see it |
| T-02-07 | Tampering | Category/location filter inputs on `/browse` | accept | Supabase JS query builder (`.eq()`, `.ilike()`) parameterizes values by construction; no string concatenation into a query, so standard SQL injection is not applicable to this code path |
</threat_model>
