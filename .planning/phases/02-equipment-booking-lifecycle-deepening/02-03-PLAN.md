<objective>
Deliver two independent vertical slices: (1) favorites — a farmer can save equipment from its detail page and retrieve it on a dedicated `/farmer/favorites` page (EQUIP-07); (2) booking terminal states — bookings can reach `completed` (owner-driven, from `approved`) and `cancelled` (farmer or owner, from `pending`/`approved`), with every transition re-verified server-side against the booking's current status (BOOK-05). Both slices depend on the live schema from Plan 02-01 (the `favorites` table and the `bookings_status_transition_guard` trigger respectively) but are otherwise unrelated feature areas bundled into one plan because each individually is too small to be its own ~50%-context plan.

Sequenced as Wave 3, after Plan 02-02 (Wave 2), because both this plan's Task 3 and Plan 02-02's Task 2 modify `src/app/(owner)/owner/dashboard/page.tsx` — running them in the same wave would risk one plan clobbering the other's edit to that file. Plan 02-02's edit (equipment list items become edit links) lands first; this plan's edit (Active Bookings section gains transition buttons) lands second, against the already-updated file.

Purpose: Completes the second half of Phase 2's goal — farmers can find equipment again later, and bookings stop dead-ending at approved/rejected.
Output: A working favorite-toggle button + favorites page; working "Mark Completed" (owner) and "Cancel" (farmer/owner) buttons on both dashboards, each backed by a service-layer `VALID_TRANSITIONS` guard and the database trigger from 02-01.
</objective>
