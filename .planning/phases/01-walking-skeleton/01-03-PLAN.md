<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Equipment creation form -> listing.service.ts | Untrusted client input (title/description/category/rate/image) crosses into a DB write + Storage upload |
| Uploaded image file -> Supabase Storage | Client-controlled file content/metadata crosses into persisted storage served back to other users (farmers viewing the listing) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-03-01 | Tampering | `owner_id` on equipment creation | mitigate | `createEquipmentAction` derives `ownerId` exclusively from the server-side authenticated session (`src/lib/supabase/server.ts`), never from form data — a forged `ownerId` field in a direct POST cannot change who owns the created listing, and the RLS `INSERT` policy from Plan 01-01 additionally enforces `owner_id = auth.uid()` at the database layer |
| T-03-02 | Tampering | Equipment category | mitigate | `createEquipmentSchema`'s `z.enum` rejects any value outside the 6 fixed categories from D-03, both for form submissions and any direct API call bypassing the UI |
| T-03-03 | Tampering | Malicious/oversized file upload bypassing the client `accept` attribute | mitigate | Server-side `imageFileSchema` re-validates MIME type and 5MB size cap inside `listing.service.ts` before calling Storage; bucket-level `allowed_mime_types`/size limit (configured in Plan 01-01 user_setup) provides a second independent enforcement layer at the Storage service itself |
| T-03-04 | Information Disclosure | Equipment browse/detail RLS | accept | Equipment listings are intentionally public to all authenticated users (any farmer must be able to browse any owner's listing) — `SELECT` policy from Plan 01-01 allows open authenticated read on `equipments`/`equipment_images`, which is the correct and intended trust boundary for a public marketplace listing, not a gap |

</threat_model>
