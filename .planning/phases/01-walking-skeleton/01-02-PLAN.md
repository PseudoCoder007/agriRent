<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Signup form input -> auth.service.ts | Untrusted client input (email/password/role) crosses into a privileged operation (creating an auth user + a role-bearing profile row) |
| Middleware role redirect vs RLS | Middleware redirect is UX-only and can be bypassed by a direct API/fetch call that skips the browser navigation — RLS remains the actual authorization boundary for data access |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-02-01 | Elevation of Privilege | Signup role selection | mitigate | `signupSchema` restricts `role` to `z.enum(["farmer","owner"])` — no arbitrary string reaches the `users` insert; role is fixed at signup and never re-editable per D-02, with no role-switching UI/endpoint built anywhere in this plan |
| T-02-02 | Spoofing | Login error messages | mitigate | `logIn()` returns a generic "Invalid email or password" rather than relaying Supabase's raw auth error text, reducing credential-enumeration signal |
| T-02-03 | Elevation of Privilege | Role read from `user_metadata` | mitigate | Every role check (signup insert, middleware, both layouts) reads exclusively from `public.users.role`; verified via grep that `user_metadata` does not appear in `middleware.ts` or either layout file |
| T-02-04 | Tampering | Direct API call bypassing middleware's role redirect | accept | Middleware/layout redirects are UX-level only by design (per ARCHITECTURE.md Pattern); the actual authorization boundary for any data read/write is the RLS policy set established in Plan 01-01, which applies regardless of which path (UI, direct fetch, future mobile client) reaches the database |

</threat_model>
