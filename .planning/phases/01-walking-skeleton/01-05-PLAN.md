<objective>
Build the booking + notification + dashboard slice: a farmer requests a booking for a date range with the price always computed server-side from the equipment's stored rate (never trusting client input), the Postgres EXCLUDE constraint is the authoritative no-double-booking guarantee (caught and translated to a friendly message, never a raw 500), an owner approves/rejects from a bare-minimum `/owner/dashboard`, a farmer sees status on `/farmer/dashboard`, and a notification row is synchronously written and awaited on every status transition, displayed as a plain list embedded in each dashboard (D-06 — no separate `/notifications` route).

Purpose: Per SKELETON.md and ARCHITECTURE.md, this is the most architecturally load-bearing slice — the state machine + conflict detection must be proven correct here, on top of the now-existing equipment listings from Plan 01-03. Per CLAUDE.md and PITFALLS.md, server-computed pricing and DB-level conflict prevention are non-negotiable trust boundaries, and Vercel Hobby gives no guaranteed post-response execution, so notification writes must be awaited before the response returns.
Output: Full booking create -> approve/reject flow with server-enforced state machine and pricing, role-scoped dashboards (D-05: separate `/owner/dashboard` and `/farmer/dashboard` routes) each embedding a plain-list notification view (D-06).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Booking request form -> booking.service.ts | Untrusted client input (equipmentId, date range) crosses into a price-computing, conflict-checked DB write |
| Owner approve/reject action -> booking.service.ts | Untrusted client action (button click, bookingId) crosses into a status-transition write that must verify both ownership and current-status preconditions server-side |
| Notification write on behalf of another user | Crosses a normal per-user RLS boundary by design — requires the service-role admin client, isolated and audited per ARCHITECTURE.md Pattern 3 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-05-01 | Tampering | Client-submitted `total_amount`/`status` on booking creation | mitigate | `createBookingSchema.strict()` causes parse failure on any extra field including `total_amount`/`status`; `booking.service.ts`'s `createBooking` computes `total_amount` exclusively from the server-fetched `equipments.rate`, never from caller input — satisfies BOOK-03 and the schema_push_requirement's explicit Zod constraint |
| T-05-02 | Tampering | Double-booking race condition | mitigate | The EXCLUDE constraint from Plan 01-01 is the authoritative guard; `createBooking` catches Postgres `23P01` specifically and translates it to a friendly "dates no longer available" message — never a raw 500, satisfying BOOK-02 |
| T-05-03 | Elevation of Privilege | Owner approving/rejecting another owner's booking | mitigate | `approveBooking`/`rejectBooking` verify `equipment.owner_id === ownerId` server-side before any status mutation, independent of UI hiding of the buttons; RLS `UPDATE` policy on `bookings` from Plan 01-01 (via the `owns_equipment()` helper) provides a second independent enforcement layer |
| T-05-04 | Tampering | Arbitrary status jump (e.g. client requesting `status: 'completed'` directly) | mitigate | `approveBooking`/`rejectBooking` hardcode their target status internally and verify the *current* status is `'pending'` before transitioning — no Server Action or service function accepts a caller-supplied target status string at all, satisfying BOOK-05's Phase-1 scope (pending -> approved/rejected only) |
| T-05-05 | Repudiation | Booking/notification write silently failing post-response | mitigate | `notificationService.createNotification` is always `await`-ed inside `booking.service.ts` before the calling function returns, never fire-and-forget — per SKELETON.md's explicit Vercel Hobby guard, satisfying NOTIF-01's "awaited before response" requirement |

</threat_model>
