<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> Supabase (anon key) | Untrusted client can issue any SELECT/INSERT/UPDATE/DELETE the anon key's RLS policies allow — RLS is the only enforcement layer here |
| Server (Next.js runtime) -> Supabase (service_role) | Fully trusted, bypasses RLS entirely — must never be reachable from client code or `NEXT_PUBLIC_*` env vars |
| Vercel serverless function -> Supabase Postgres | Connection-level trust boundary — must use pooled (6543) connection, not direct (5432), to avoid connection exhaustion under concurrent invocations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-------------------|
| T-01-01 | Elevation of Privilege | RLS policy on `equipments`/`bookings` | mitigate | Role/ownership checks routed exclusively through `SECURITY DEFINER LANGUAGE plpgsql` helper functions (`is_owner()`, `owns_equipment()`), never a direct cross-table subquery inside a policy — closes the infinite-recursion/accidental-permissive-fallback failure mode from PITFALLS.md Pitfall 1 |
| T-01-02 | Tampering | `bookings` table double-insert under concurrency | mitigate | Postgres `EXCLUDE USING gist` constraint with `btree_gist`, scoped to `status IN ('pending','approved')` — database-level guarantee, not application-level check-then-insert |
| T-01-03 | Information Disclosure | `service_role` key exposure | mitigate | Key lives only in `src/lib/supabase/admin.ts`, read from `process.env.SUPABASE_SERVICE_ROLE_KEY` (no `NEXT_PUBLIC_` prefix), file carries a runtime guard throwing if `typeof window !== 'undefined'`, never imported into a Client Component |
| T-01-04 | Elevation of Privilege | Role stored in `user_metadata` (client-editable) | mitigate | Role lives in the server-controlled `public.users` table, populated at signup server-side, never read from `auth.users.user_metadata` inside any RLS policy |
| T-01-05 | Denial of Service | Direct (non-pooled) Postgres connection under concurrent serverless invocations | mitigate | `DATABASE_URL` documented in `.env.local.example` as the Supavisor transaction-mode pooled string (port 6543); user_setup instructs sourcing the pooled connection string explicitly, not the default direct one |
| T-01-SC | Tampering | npm package installs (`@supabase/*`, `openai`, `zod`, etc.) | accept | All packages in this task are well-known, high-download-count, official-vendor packages (`@supabase/supabase-js`, `@supabase/ssr`, `openai`, `zod`, `react-hook-form`) already named and version-pinned in research/STACK.md with npm registry verification — no `[ASSUMED]`/`[SUS]` packages introduced; no blocking human-verify checkpoint needed for this plan's installs |

</threat_model>
